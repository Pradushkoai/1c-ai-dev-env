# /1c-query-opt — Оптимизация запросов 1С

Продвинутые паттерны оптимизации запросов за пределами базовых правил.

## Когда использовать

- Сложная многошаговая обработка данных
- Оптимизация JOIN и подзапросов
- Работа с отчётами СКД
- Обработка больших объёмов данных
- Медленные запросы и диагностика производительности

---

## 1. JOIN vs Подзапрос {#join-vs-subquery}

### Паттерн
Используйте JOIN вместо подзапроса в SELECT — обычно быстрее.

### Пример

```bsl
// ПРАВИЛЬНО: JOIN (обычно быстрее)
"ВЫБРАТЬ
|       Заказы.Ссылка, Контрагенты.ИНН
|ИЗ Документ.ЗаказКлиента КАК Заказы
|       ЛЕВОЕ СОЕДИНЕНИЕ Справочник.Контрагенты КАК Контрагенты
|       ПО Заказы.Контрагент = Контрагенты.Ссылка"

// ПЛОХО: Подзапрос в SELECT (N+1)
"ВЫБРАТЬ
|       Заказы.Ссылка,
|       (ВЫБРАТЬ К.ИНН ИЗ Справочник.Контрагенты КАК К
|        ГДЕ К.Ссылка = Заказы.Контрагент) КАК ИНН
|ИЗ Документ.ЗаказКлиента КАК Заказы"
```

### Связанные правила
- BSL LS: `JoinWithSubQuery` — JOIN с подзапросом (антипаттерн)

## 2. Никогда не JOIN с подзапросами — используй временные таблицы {#temp-table-vs-join-subquery}

### Паттерн
JOIN с подзапросом (с GROUP BY) — крайне медленно. Используйте временную таблицу.

### Пример

```bsl
// ПЛОХО: JOIN с подзапросом
"... ЛЕВОЕ СОЕДИНЕНИЕ (ВЫБРАТЬ ... СГРУППИРОВАТЬ ПО ...) КАК Итоги ..."

// ПРАВИЛЬНО: Временная таблица + JOIN
"ВЫБРАТЬ ... ПОМЕСТИТЬ ВТИтоги ... ИНДЕКСИРОВАТЬ ПО ...;
ВЫБРАТЬ ... ИЗ ... ЛЕВОЕ СОЕДИНЕНИЕ ВТИтоги ..."
```

## 3. Параметры виртуальных таблиц {#virtual-table-params}

### Паттерн
Фильтруйте данные В ПАРАМЕТРАХ виртуальной таблицы, а не в WHERE.

### Пример

```bsl
// ПРАВИЛЬНО: Фильтр В ПАРАМЕТРАХ виртуальной таблицы
"ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки(
|       &Дата, Номенклатура В (&Список)) КАК Остатки"

// ПЛОХО: Виртуальная таблица + ГДЕ
"ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки(&Дата) КАК Остатки
|ГДЕ Остатки.Номенклатура В (&Список)"
```

### Связанные правила
- BSL LS: `JoinWithVirtualTable` — JOIN с виртуальной таблицей (антипаттерн)

## 4. Составные типы — ВЫРАЗИТЬ {#cast-composite-types}

### Паттерн
При разыменовании составного типа используйте ВЫРАЗИТЬ к конкретному типу.

### Пример

```bsl
// ПЛОХО: Разыменование составного типа (запрос по ВСЕМ типам)
"ВЫБРАТЬ ТоварыНаСкладах.Регистратор.Дата"

// ПРАВИЛЬНО: ВЫРАЗИТЬ к конкретному типу
"ВЫБРАТЬ ВЫРАЗИТЬ(ТоварыНаСкладах.Регистратор
|       КАК Документ.ПоступлениеТоваровУслуг).Дата"
```

## 5. ПРЕДСТАВЛЕНИЕ для отображения {#presentation-function}

### Паттерн
Используйте ПРЕДСТАВЛЕНИЕ() вместо обращения через точку для отображения.

### Пример

```bsl
// ПЛОХО: Доп. JOIN через точку
"ВЫБРАТЬ ТоварыНаСкладах.Склад.Наименование"

// ПРАВИЛЬНО: Без доп. JOIN
"ВЫБРАТЬ ПРЕДСТАВЛЕНИЕ(ТоварыНаСкладах.Склад) КАК Склад"
```

## 6. ИЛИ в ГДЕ → ОБЪЕДИНИТЬ ВСЕ {#or-to-union-all}

### Паттерн
ИЛИ в WHERE блокирует использование индексов. Замените на ОБЪЕДИНИТЬ ВСЕ.

### Пример

```bsl
// ПЛОХО: ИЛИ блокирует индексы
"ГДЕ Товары.Артикул = &Артикул ИЛИ Товары.Код = &Код"

// ПРАВИЛЬНО: Два запроса с ОБЪЕДИНИТЬ ВСЕ
"ВЫБРАТЬ ... ГДЕ Товары.Артикул = &Артикул
|ОБЪЕДИНИТЬ ВСЕ
|ВЫБРАТЬ ... ГДЕ Товары.Код = &Код"
```

### Связанные правила
- BSL LS: `LogicalOrInTheWhereSectionOfQuery` — ИЛИ в WHERE (проблема производительности)

## 7. ОБЪЕДИНИТЬ vs ОБЪЕДИНИТЬ ВСЕ {#union-vs-union-all}

### Паттерн
Всегда `ОБЪЕДИНИТЬ ВСЕ` если не нужно удалять дубликаты (быстрее — нет группировки).

### Пример

```bsl
// ПРАВИЛЬНО: ОБЪЕДИНИТЬ ВСЕ — без удаления дубликатов (быстрее)
"ВЫБРАТЬ ... ОБЪЕДИНИТЬ ВСЕ ВЫБРАТЬ ..."

// МЕДЛЕННЕЕ: ОБЪЕДИНИТЬ — удаляет дубликаты (дополнительная группировка)
"ВЫБРАТЬ ... ОБЪЕДИНИТЬ ВЫБРАТЬ ..."
```

## 8. Виртуальные таблицы — сначала во временную {#virtual-table-to-temp}

### Паттерн
Сначала извлеките данные из виртуальной таблицы во временную, потом JOIN.

### Пример

```bsl
// ПРАВИЛЬНО: Сначала извлечь, потом JOIN
"ВЫБРАТЬ ... ПОМЕСТИТЬ ВТОстатки
|ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки(&Дата,) ИНДЕКСИРОВАТЬ ПО Номенклатура;
|ВЫБРАТЬ ... ИЗ Справочник.Номенклатура ЛЕВОЕ СОЕДИНЕНИЕ ВТОстатки ..."
```

## 9. Индексы {#indexes}

### Паттерн
Индекс должен содержать все поля условия, поля должны быть в начале и последовательны.

### Правила
1. Индекс должен содержать **все поля** условия
2. Поля должны быть **в начале** индекса
3. Поля должны быть **последовательными** (без пропусков)
4. `ИНДЕКСИРОВАТЬ ПО` для временных таблиц при JOIN

### Пример

```bsl
// ПРАВИЛЬНО: Индекс по полям условия
"ВЫБРАТЬ ... ПОМЕСТИТЬ ВТДанные ИНДЕКСИРОВАТЬ ПО Номенклатура, Склад ...
|ВЫБРАТЬ ... ИЗ ВТДанные ГДЕ Номенклатура = &Ном И Склад = &Скл"
```

## 10. ПОДОБНО с % в начале — full scan {#like-optimization}

### Паттерн
ПОДОБНО с `%` в начале строки вызывает full table scan. Используйте `%` только в конце.

### Пример

```bsl
// ПЛОХО: % в начале — full scan (не использует индекс)
"ГДЕ Товары.Наименование ПОДОБНО \"%текст\""

// ПРАВИЛЬНО: % только в конце — использует индекс
"ГДЕ Товары.Наименование ПОДОБНО \"текст%\""
```

### Связанные правила
- Наше Q005: ПОДОБНО '%...' — full scan
- BSL LS: `UsingLikeInQuery` — использование ПОДОБНО (предупреждение)
- BSL LS: `IncorrectUseLikeInQuery` — неправильное использование ПОДОБНО

## 11. Запрос в цикле — N+1 проблема {#no-query-in-loop}

### Паттерн
Никогда не выполняйте запросы в цикле. Используйте один запрос с JOIN.

### Пример

```bsl
// ПЛОХО: Запрос в цикле (N+1)
Для Каждого Стр Из Таблицы Цикл
    Запрос = Новый Запрос("ВЫБРАТЬ ... ГДЕ Код = &Код");
    Запрос.УстановитьПараметр("Код", Стр.Код);
    Результат = Запрос.Выполнить();
КонецЦикла;

// ПРАВИЛЬНО: Один запрос с фильтром по массиву
Запрос = Новый Запрос("ВЫБРАТЬ ... ГДЕ Код В (&МассивКодов)");
Запрос.УстановитьПараметр("МассивКодов", Таблица.ВыгрузитьКолонку("Код"));
```

### Связанные правила
- Наше Q010: Временная таблица без индекса
- BSL LS: `CreateQueryInCycle` — запрос в цикле

## 12. ВЫБРАТЬ * — выбирать только нужные поля {#no-select-star}

### Паттерн
Никогда не используйте `ВЫБРАТЬ *` — выбираются все поля, включая ненужные.

### Пример

```bsl
// ПЛОХО: ВЫБРАТЬ * — выбираются все поля
"ВЫБРАТЬ * ИЗ Справочник.Номенклатура"

// ПРАВИЛЬНО: Только нужные поля
"ВЫБРАТЬ Ссылка, Код, Наименование ИЗ Справочник.Номенклатура"
```

### Связанные правила
- Наше Q001: ВЫБРАТЬ * — неоптимальный запрос

## 13. Параметризованные запросы — без SQL-инъекций {#parameterized-queries}

### Паттерн
Всегда используйте параметры (`&Параметр`), никогда не конкатенируйте строки.

### Пример

```bsl
// ПЛОХО: Конкатенация строк — SQL-инъекция
ТекстЗапроса = "ВЫБРАТЬ ... ГДЕ Код = """ + Код + """";

// ПРАВИЛЬНО: Параметризованный запрос
ТекстЗапроса = "ВЫБРАТЬ ... ГДЕ Код = &Код";
Запрос.УстановитьПараметр("Код", Код);
```

### Связанные правила
- Наше Q003: Запрос без параметров
- Наше Q004: Конкатенация строк в запросе (SQL-инъекция)
- SEC001: SQL-инъекция (security_auditor)

## 14. SELECT TOP без ORDER BY — некорректный результат {#select-top-with-order}

### Паттерн
SELECT TOP без ORDER BY возвращает случайные строки. Всегда добавляйте ORDER BY.

### Пример

```bsl
// ПЛОХО: TOP без ORDER BY — какие строки вернутся?
"ВЫБРАТЬ ПЕРВЫЕ 10 ... ИЗ ..."

// ПРАВИЛЬНО: TOP с ORDER BY
"ВЫБРАТЬ ПЕРВЫЕ 10 ... ИЗ ... УПОРЯДОЧИТЬ ПО Дата УБЫВ"
```

### Связанные правила
- BSL LS: `SelectTopWithoutOrderBy` — SELECT TOP без ORDER BY

---

## 15. Full Outer Join — проблемное соединение {#full-outer-join}

### Паттерн
Полное внешнее соединение (FULL JOIN) часто медленнее и сложнее для понимания.
Рассмотрите альтернативы.

### Пример

```bsl
// ПЛОХО: FULL OUTER JOIN — медленно и сложно
"ВЫБРАТЬ ... ИЗ А ПОЛНОЕ СОЕДИНЕНИЕ Б ПО А.Ключ = Б.Ключ"

// ПРАВИЛЬНО: Два LEFT JOIN через ОБЪЕДИНИТЬ ВСЕ
"ВЫБРАТЬ ... ИЗ А ЛЕВОЕ СОЕДИНЕНИЕ Б ПО А.Ключ = Б.Ключ
|ОБЪЕДИНИТЬ ВСЕ
|ВЫБРАТЬ ... ИЗ Б ЛЕВОЕ СОЕДИНЕНИЕ А ПО Б.Ключ = А.Ключ
|ГДЕ А.Ключ ЕСТЬ NULL"
```

### Связанные правила
- BSL LS: `FullOuterJoinQuery` — полное внешнее соединение
