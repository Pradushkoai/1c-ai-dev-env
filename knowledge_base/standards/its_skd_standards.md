# Стандарты 1С: СКД (Схема Компоновки Данных)

**Источник:** [v8std.ru](https://v8std.ru) | [ITS 1С](https://its.1c.ru/db/v8std)  
**Связанные инструменты:** `check_skd_quality`, `get_skd_schema`, `skd_trace`, `dsl_compile_skd`, `generate_report`, `solve_check`

> Полный каталог стандартов 1С по СКД (Схеме Компоновки Данных): структура, параметры, наборы данных, отборы, условное оформление, варианты отчётов, динамические списки. Стандарты #std407, #std670–#std676, #std710, #std732, #std768 и смежные.

---

## Общие сведения

СКД (Схема Компоновки Данных) — декларативный механизм платформы 1С:Предприятие для построения отчётов. Состоит из:
- **Схемы** — наборы данных, параметры, поля, ресурсы
- **Настроек** — отборы, группировки, условное оформление, порядок
- **Вариантов** — сохранённые настройки для разных аналитик
- **Макетов** — оформление вывода

Большинство стандартов 1С по отчётам (#std670–#std676) формулируются именно в терминах СКД.

## #std407: Применение параметров отчёта в СКД

**ITS:** https://its.1c.ru/db/v8std/content/407/hdoc | **v8std:** https://v8std.ru/std/407/

### Правила
1. Избегайте параметров отчёта, если можно обойтись обычным управлением элементами отбора. Не задавайте условие `ГДЕ Организация = &Организация` — дайте пользователю управлять отбором по полю `Организация`.
2. Избегайте обязательных параметров, чтобы пользователь не получал ошибку «Не задан параметр отчёта» при отключении параметров.
3. Для необязательных параметров используйте расширение языка запросов СКД вида `{ГДЕ ...}` — при отключении параметра пользователем фрагмент условия автоматически исключается.

### Пример

```bsl
// Необязательные параметры: если пользователь их отключит — условие исчезнет
ВЫБРАТЬ
    Сведения.Номенклатура КАК Номенклатура,
    Сведения.Сумма КАК Сумма
ИЗ
    РегистрНакопления.Продажи КАК Сведения
{ГДЕ
    Сведения.Период >= &ДатаНачала,
    Сведения.Период <= &ДатаОкончания,
    Сведения.Контрагент = &Контрагент}
```

### Связанные правила
- BSL LS: `AssignAliasFieldsInQuery` (через #std437)
- Наши правила: проверка параметров СКД SKD006

## #std670: Содержание отчёта

**ITS:** https://its.1c.ru/db/v8std/content/670/hdoc | **v8std:** https://v8std.ru/std/670/

### Правила
1. Состав показателей соответствует назначению отчёта (цели анализа). В отчёт включаются только показатели, связанные общей целью.
2. В отчётах **без** СКД рекомендуется обеспечивать расшифровку всех выводимых показателей.
3. Отчёты не должны содержать данные, помеченные на удаление (либо выделять их явно).
4. Для сравнительного анализа валюта показателя должна быть единой в пределах отчёта; при необходимости выводите одновременно исходную и приведённую валюту.
5. Различайте **контекстные** (запускаются из формы объекта, с предустановленными отборами/параметрами) и **неконтекстные** отчёты. Контекстные варианты не должны быть доступны в общем списке вариантов; имя контекстного варианта должно содержать слово «Контекст».

### Пример именования варианта

```
ДвиженияСерийНоменклатурыКонтекст   // для отчёта «Движения серий товаров»
```

## #std671: Варианты отчётов

**ITS:** https://its.1c.ru/db/v8std/content/671/hdoc | **v8std:** https://v8std.ru/std/671/

### Правила
1. Создавайте варианты отчёта, когда один источник данных нужно анализировать в разных группировках, сортировках, отборах, наборах показателей (один отчёт — несколько вариантов).
2. Если вариант — рабочее место или используется очень часто, оформляйте его отдельным отчётом.
3. Для разных подсистем одного раздела создавайте отдельные варианты с разными отборами, а не один общий.
4. При использовании БСП обязательно заполняйте описание варианта (назначение + состав анализируемых данных).

### Пример (правильно — два варианта с разными отборами)

```
Валовая прибыль по поставщикам (опт)      // отбор по оптовым продажам
Валовая прибыль по поставщикам (розница)  // отбор по розничным продажам
```

### Антипаттерн
Один вариант «Универсальный» со всеми отборами — пользователь не понимает, что он видит.

## #std672: Поля периодов

**ITS:** https://its.1c.ru/db/v8std/content/672/hdoc | **v8std:** https://v8std.ru/std/672/

### Правила
1. При настройке схемы СКД придерживайтесь единых имён полей периодов: `Период.Год`, `Период.Квартал`, `Период.Месяц`, `Период.День`, `Период.Час` (синоним «Период, год», «Период, квартал»…).
2. Применяйте стандартные форматные строки:
   - Год — `Л=ru; ДФ='yyyy "г."'`
   - Квартал — `Л=ru; ДФ='к "квартал" гггг "г."'`
   - Месяц — `Л=ru; ДФ='MMMM гггг "г."'`
   - День — `Л=ru; ДФ='дд.ММ.гггг'`
3. Поле `ПериодСекунда` — служебное: ограничьте его использование (не для выбора/отбора/группировки пользователем).
4. Для поля `Регистратор` укажите выражение представления `Регистратор` и выражение упорядочивания `ПериодСекунда возр, Регистратор возр` — это нужно для корректного упорядочивания при расшифровке по регистратору.

### Пример (настройка поля периода в схеме)

```
Поле:        Период.Месяц
Синоним:     Период, месяц
Формат:      Л=ru; ДФ='MMMM гггг "г."'
```

## #std673: Пользовательские настройки

**ITS:** https://its.1c.ru/db/v8std/content/673/hdoc | **v8std:** https://v8std.ru/std/673/

### Правила
1. Избегайте неоднозначных наименований настроек. Для параметров, без которых запрос СКД не выполнится, ставьте флажок «Запрещать незаполненные значения» и режим использования «Всегда».
2. **Быстрые пользовательские настройки** — не более 5–7. Включайте в них частотные параметры (контекстное меню не позволяет применять параметры «на лету») и отборы по наиболее важным группировкам.
3. В быстрые настройки контекстных отчётов не включайте отборы по полям, которые уже установлены через параметры вызова.
4. В обычные настройки включайте: отборы по невыведенным реквизитам, отборы по числовым показателям, «Выбранные поля», «Отборы», «Сортировка», «Условное оформление», «Группировка».

### Пример

Обязательный параметр «Период = Этот месяц» для отчёта «Выручка и себестоимость продаж».

## #std674: Заголовок отчёта

**ITS:** https://its.1c.ru/db/v8std/content/674/hdoc | **v8std:** https://v8std.ru/std/674/

### Правила
1. Заголовок отчёта чётко отражает цель анализа; для каждого варианта заголовок заполняется и соответствует наименованию варианта (чтобы отчёт находился по напечатанному заголовку).
2. Уточнение по группировке пишите во множественном числе без разделителей: «Анализ расчетов с клиентами по договорам» (а не «…(по договорам)»).
3. **Запрещено** называть вариант «Основной» — оно не раскрывает смысл.
4. Избегайте слова «отчёт» в синониме и заголовках.

### Проверки
- EDT v8-code-style: `#v8cs:data-composition-variant-name-default`

## #std675: Отчёты вида «диаграмма»

**ITS:** https://its.1c.ru/db/v8std/content/675/hdoc | **v8std:** https://v8std.ru/std/675/

### Правила
1. Диаграммы применяйте только при небольшом объёме данных. Выбирайте тип по цели:
   - динамика → график/гистограмма
   - распределение → вертикальная гистограмма
   - структура → круговая (до 7–10 компонент) или гистограмма-рейтинг
2. Избегайте объёмных (3D) диаграмм — они искажают информацию.
3. Основной цвет гистограмм/линий динамики — элемент стиля «Диаграмма» (RGB 70,130,180); прогнозные значения — «Прогноз» (RGB 199,21,133).
4. Не используйте оттенки красного/зелёного для серий, если они не отражают негативные/позитивные факты.

## #std676: Отчёты вида «таблица», «список»

**ITS:** https://its.1c.ru/db/v8std/content/676/hdoc | **v8std:** https://v8std.ru/std/676/

### Правила
1. Уровней вложенности группировок — **не более 3**; вложенность отражает связь «от общего к частному».
2. Группировку по периоду располагайте в колонках с упорядочиванием по возрастанию.
3. Показатели в колонках — по важности слева направо или в порядке логики вычислений; в строках — если нужно сравнивать показатели в динамике.
4. Сортировка по умолчанию — по назначению отчёта (самые важные данные наверху); сортировка применяется только к видимым объектам анализа.

## #std710: Условное оформление в формах

**ITS:** https://its.1c.ru/db/v8std/content/710/hdoc | **v8std:** https://v8std.ru/std/710/

### Правила
1. Не используйте условное оформление, чтобы скрывать строки целиком — замедляет веб-клиент и искажает таблицу.
2. Если задачу решает и условное оформление динамического списка, и формы — выбирайте оформление списка (ускоряет открытие формы).
3. Настройку выполняйте **в коде** при создании формы и далее не модифицируйте: однотипные настройки выносятся в общие модули, легче объединять конфигурации, ошибки выявляются при синтаксическом контроле.
4. Минимизируйте строковые константы в коде оформления — используйте типизированные конструкции (`Перечисления…`, разыменование).

### Пример

```bsl
&НаСервере
Процедура УстановитьУсловноеОформление()
    УсловноеОформление.Элементы.Очистить();
    Элемент = УсловноеОформление.Элементы.Добавить();

    ПолеЭлемента = Элемент.Поля.Элементы.Добавить();
    ПолеЭлемента.Поле = Новый ПолеКомпоновкиДанных(Элементы.ТоварыУпаковка.Имя);

    ГруппаОтбора = Элемент.Отбор.Элементы.Добавить(Тип("ГруппаЭлементовОтбораКомпоновкиДанных"));
    ГруппаОтбора.ТипГруппы = ТипГруппыЭлементовОтбораКомпоновкиДанных.ГруппаИли;

    Отбор = ГруппаОтбора.Элементы.Добавить(Тип("ЭлементОтбораКомпоновкиДанных"));
    Отбор.ЛевоеЗначение = Новый ПолеКомпоновкиДанных("АдресноеХранение");
    Отбор.ВидСравнения  = ВидСравненияКомпоновкиДанных.Равно;
    Отбор.ПравоеЗначение = Ложь;

    Элемент.Оформление.УстановитьЗначениеПараметра("ОтметкаНезаполненного", Ложь);
КонецПроцедуры
```

### Проверки
- EDT v8-code-style: `#v8cs:data-composition-conditional-appearance-use`
- АПК: `#acc:386`, `#acc:387`

## #std732: Запросы в динамических списках

**ITS:** https://its.1c.ru/db/v8std/content/732/hdoc | **v8std:** https://v8std.ru/std/732/

### Правила
1. Запросы динамических списков делайте максимально простыми; сложные расчёты выносите в регистры сведений, обновляемые при проведении или регламентным заданием.
2. Предпочитайте режим «Динамическое считывание данных включено» — запрос выбирает количество записей, близкое к числу видимых строк.
3. **Не используйте** `РАЗЛИЧНЫЕ` и `СГРУППИРОВАТЬ ПО`, `ВЫБОР` в `ГДЕ`/условиях соединения, сортировку по полю из `ВЫБОР`.
4. Соединяйте с небольшим числом реальных таблиц (ориентир — не более 4); не соединяйте с вложенными запросами и виртуальными таблицами (кроме `СрезПоследних`/`СрезПервых` при малом числе записей).

### Пример (правильно — состояние хранится в регистре)

```bsl
// Состояние отгрузки вычислено заранее в регистре сведений
ВЫБРАТЬ
    Распоряжения.Ссылка КАК Распоряжение,
    СостоянияОтгрузки.Состояние КАК Состояние
ИЗ
    Документ.ЗаказКлиента КАК Распоряжения
    ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.СостоянияОтгрузки.СрезПоследних КАК СостоянияОтгрузки
        ПО Распоряжения.Ссылка = СостоянияОтгрузки.Заказ
```

## #std768: Программное переопределение текстов запросов динамических списков

**ITS:** https://its.1c.ru/db/v8std/content/768/hdoc | **v8std:** https://v8std.ru/std/768/

### Правила
1. Переопределяйте текст запроса динамического списка в `ПриСозданииНаСервере`, формируя его из типовых фрагментов.
2. Каждый фрагмент должен открываться в конструкторе запросов — для быстрой проверки синтаксиса и сопровождения.
3. Комментируйте все этапы программной сборки текста.

## #std489: Ограничения при использовании динамических списков

**ITS:** https://its.1c.ru/db/v8std/content/489/hdoc | **v8std:** https://v8std.ru/std/489/

### Правила
1. Группировки в динамических списках (построены на СКД) используйте только при заведомо небольшом количестве записей; в больших списках группировка — только по индексированным полям.
2. Многоуровневые группировки допустимы, если поле первой группировки индексировано и достаточно селективно.
3. Не делайте группировки по характеристикам объекта (используется `ЛЕВОЕ СОЕДИНЕНИЕ` с таблицей характеристик — отбор неэффективен даже с индексом).
4. В иерархических списках не ставьте `НачальноеОтображениеДерева = РаскрыватьВсеУровни` — критично снижает скорость. Используйте `НеРаскрывать` / `РаскрыватьВерхнийУровень`.

## #std702: Реквизит «Ссылка» и признак «Использовать всегда» в динамических списках

**ITS:** https://its.1c.ru/db/v8std/content/702/hdoc | **v8std:** https://v8std.ru/std/702/

### Правила
1. В списках ссылочных объектов добавляйте поле `Ссылка` с отключенной пользовательской видимостью. Для механизмов характеристик и команд печати БСП `Ссылка` обязательна.
2. Признак «Использовать всегда» ставьте только когда значение колонки обязательно для алгоритма (например, колонка `Ссылка` для команды «Печать»).
3. Не ставьте «Использовать всегда» для каждой колонки — это лишняя нагрузка.

### Проверки
- EDT: `#v8cs:form-list-field-ref-not-added`, `#v8cs:form-list-ref-use-always-flag-disabled`, `#v8cs:form-list-ref-user-visibility-enabled`
- АПК: `#acc:148`, `#acc:149`, `#acc:160`

## #std399: Запрет редактирования полей таблицы по условию

**ITS:** https://its.1c.ru/db/v8std/content/399/hdoc | **v8std:** https://v8std.ru/std/399/

### Правила
1. Если ввод значения не имеет смысла до заполнения других полей — используйте предупреждение при редактировании (свойства `ОтображениеПредупрежденияПриРедактировании` и `ПредупреждениеПриРедактировании`).
2. Управляйте отображением из кода при активизации строки и при изменении влияющих значений.
3. Сохраняйте пользователю возможность изменить/очистить уже введённое «не по правилам» значение.

### Пример

```bsl
&НаКлиенте
Процедура УстановитьПредупреждениеПриРедактировании()
    ТребуетсяПредупреждение =
        Не ЗначениеЗаполнено(Элементы.Затраты.ТекущиеДанные.СтатьяЗатрат)
        И Не ЗначениеЗаполнено(Элементы.Затраты.ТекущиеДанные.АналитикаСтатьиЗатрат);

    Элементы.ЗатратыАналитикаСтатьяЗатрат.ОтображениеПредупрежденияПриРедактировании = ?(
        ТребуетсяПредупреждение,
        ОтображениеПредупрежденияПриРедактировании.Отображать,
        ОтображениеПредупрежденияПриРедактировании.НеОтображать);
КонецПроцедуры
```

## #std762: Локализация запросов в СКД и динамических списках

**ITS:** https://its.1c.ru/db/v8std/content/762/hdoc | **v8std:** https://v8std.ru/std/762/

### Правила
1. Тексты запросов в наборах данных СКД и динамических списках должны быть локализуемыми: строковые литералы — через `НСтр`, не захардкоженные русские тексты.
2. Поля-периоды, форматы дат/чисел — в соответствии с #std763 (форматирование).
3. Имена полей и синонимы — вынесены для перевода.
