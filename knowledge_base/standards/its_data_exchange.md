# Стандарты 1С: Обмен данными

**Источник:** [v8std.ru](https://v8std.ru) | [ITS 1С](https://its.1c.ru/db/v8std)  
**Связанные инструменты:** `check_transactions`, `solve_check`, `check_standards`, `analyze_bsl`

> Полный каталог стандартов 1С по обмену данными: планы обмена, формат EnterpriseData, правила конвертации, XML/XDTO-сериализация, признак `ОбменДанными.Загрузка`. Стандарты #std701, #std771, #std773, #std637, #std464, #std465, #std752.

---

## Общие сведения

Обмен данными в 1С реализуется через:
- **Планы обмена** (РИБ, синхронизация между одинаковыми/разными конфигурациями) с регистрацией изменений по узлам
- **Формат EnterpriseData** (канонический XML-формат БСП для обмена между конфигурациями на БСП)
- **Правила конвертации** (Конвертация данных 2.0 / 3.0, инструмент «Конвертация данных»)
- **XML/XDTO-сериализацию** и ручной обмен через XML/JSON
- признак **`ОбменДанными.Загрузка`** для записи объектов «как есть» без бизнес-логики

## #std701: Разработка планов обмена с отборами

**ITS:** https://its.1c.ru/db/v8std/content/701/hdoc | **v8std:** https://v8std.ru/std/701/

### Правила
1. Узлы-получатели определяйте в обработчиках `ПередЗаписью` (#std464) и `ПередУдалением` (#std752). При использовании подсистемы «Обмен данными» БСП логику регистрации задают декларативно в правилах регистрации объектов (ПРО) через «Конвертацию данных».
2. Учитывайте: данные из сообщения загружаются в произвольном порядке; логика регистрации выполняется и при загрузке (в транзакции записи, `ОбменДанными.Загрузка` игнорируется) — любая ошибка аварийно завершает загрузку.
3. Обращение к связанным таблицам «через точку» → неявные соединения и снижение производительности.
4. **Самодостаточность данных:** таблицы обмена должны содержать все данные для логики регистрации; обращайтесь только к основной таблице. Если нельзя — исключайте вторичные данные из обмена или регистрируйте изменения всех связанных таблиц.

### Пример (самодостаточные таблицы)

Документ и регистр накопления с движениями — оба содержат ссылку на организацию, логика регистрации срабатывает отдельно для каждой таблицы.

### Антипаттерн

```bsl
// ПЛОХО: обращение через точку к связанной таблице в обработчике регистрации
Процедура ПередЗаписью(Отказ)
    Если ОбменДанными.Загрузка Тогда
        Возврат;
    КонецЕсли;
    
    // Регистрация зависит от организации контрагента (через точку!)
    Организация = ЭтотОбъект.Контрагент.Организация;  // НЕЯВНОЕ СОЕДИНЕНИЕ
    МассивУзлов = ОбменДаннымиСервер.УзлыПоОрганизации(Организация);
    Для Каждого Узел Из МассивУзлов Цикл
        ПланыОбмена.ЗарегистрироватьИзменения(Узел, Ссылка);
    КонецЦикла;
КонецПроцедуры

// ПРАВИЛЬНО: организация хранится в самом документе
Процедура ПередЗаписью(Отказ)
    Если ОбменДанными.Загрузка Тогда
        Возврат;
    КонецЕсли;
    
    МассивУзлов = ОбменДаннымиСервер.УзлыПоОрганизации(ЭтотОбъект.Организация);
    Для Каждого Узел Из МассивУзлов Цикл
        ПланыОбмена.ЗарегистрироватьИзменения(Узел, Ссылка);
    КонецЦикла;
КонецПроцедуры
```

## #std771: Интеграция через формат EnterpriseData

**ITS:** https://its.1c.ru/db/v8std/content/771/hdoc | **v8std:** https://v8std.ru/std/771/

### Правила
1. Переходы между конфигурациями — на основе правил конвертации; обмен между конфигурациями на БСП — на формате **EnterpriseData**. Новые обмены на правилах конвертации не разрабатывайте (исключение — бюджетные решения вне охвата EnterpriseData).
2. Актуальные версии берите из последней опубликованной БСП: объекты `Пакет XDTO` с именами `EnterpriseData_{X}_{Y}_{Z}` (Major/Minor). Поддерживайте все актуальные версии для асинхронного выпуска/перехода.
3. Запрещено выпускать версию конфигурации, не поддерживающую какую-либо версию формата, поддерживаемую во встроенной БСП.
4. **Не используйте `AdditionalInfo`** для передачи сведений между базами в типовых обменах — расширяйте формат согласованными изменениями. `AdditionalInfo` — только для срочного законодательства/обхода критических ошибок.

### Проверки
- АПК: `#acc:518`, `#acc:519`, `#acc:520`

## #std773: Признак `ОбменДанными.Загрузка` в обработчиках событий объекта

**ITS:** https://its.1c.ru/db/v8std/content/773/hdoc | **v8std:** https://v8std.ru/std/773/

### Правила
1. В `ПередЗаписью`, `ПриЗаписи`, `ПередУдалением` сначала проверяйте `ОбменДанными.Загрузка` и при загрузке — `Возврат`. Объект загружается «как есть», без бизнес-логики.
2. Исключение — обмен, регистрирующий изменения для выгрузки на другие узлы. В БСП для отключения регистрации добавляйте доп. свойство `ОтключитьМеханизмРегистрацииОбъектов`.
3. Требование действует и для подписок на события.
4. Вызывающий код, установивший `ОбменДанными.Загрузка = Истина`, берёт на себя ответственность за целостность данных. Дозаполнение выносите в экспортные процедуры объекта и вызывайте до записи в режиме обмена.

### Пример

```bsl
Процедура ПередЗаписью(Отказ)
    Если ОбменДанными.Загрузка Тогда
        Возврат;
    КонецЕсли;
    // Код обработчика бизнес-логики
КонецПроцедуры

// Загрузка объекта без бизнес-логики
Объект.ОбменДанными.Загрузка = Истина;
Объект.Записать();

// БСП: отключить регистрацию при загрузке
Объект.ОбменДанными.Загрузка = Истина;
Объект.ДополнительныеСвойства.Вставить("ОтключитьМеханизмРегистрацииОбъектов");
Объект.Записать();
```

### Диагностики BSL LS
- `DataExchangeLoading`
- EDT: `#v8cs:data-exchange-load`
- АПК: `#acc:75`

## #std637: Настройка обмена для классификаторов между разными ИБ

**ITS:** https://its.1c.ru/db/v8std/content/637/hdoc | **v8std:** https://v8std.ru/std/637/

### Правила
1. Классификаторы (справочники и регистры сведений с условно-постоянной информацией: банки, страны, валюты, ОКЕИ, адресный) при обмене через «Конвертацию данных» сопоставляйте сначала по уникальным идентификаторам ссылок, затем — по полям поиска.
2. В ПКО классификатора установите признаки «Искать объект приёмника по внутреннему идентификатору» и «Продолжить поиск по полям поиска»; задайте ПКС-поля поиска (например, БИК для банков).
3. Классификаторы в виде наборов записей (адресный) **не должны** участвовать в обмене — заполняйте и поддерживайте отдельно в каждой базе.

## #std464: Обработчик события `ПередЗаписью` (в контексте обмена)

**ITS:** https://its.1c.ru/db/v8std/content/464/hdoc | **v8std:** https://v8std.ru/std/464/

### Правила
1. В `ПередЗаписью` реализуют логику регистрации изменений на узлах плана обмена (#std701).
2. Сначала проверяется `ОбменДанными.Загрузка` (#std773).
3. Не выполнять ресурсоёмких запросов к связанным таблицам «через точку» — снижает производительность обмена.

## Обобщённая практика: План обмена и Узлы

> Основано на #std701 и документации платформы/БСП «Обмен данными».

### Структура плана обмена
1. В плане обмена задаются **состав** (авторегистрация) и **узлы**. Для обмена с отбором авторегистрацию отключают и регистрируют изменения программно в `ПередЗаписью`/`ПередУдалением`.
2. Каждый узел имеет уникальный `Код` (= `НомерСообщения`/идентификатор узла). Центральный узел РИБ помечается признаком «Этот узел — центральный» (только в РИБ).
3. Регистрация изменений: `ПланыОбмена.ЗарегистрироватьИзменения(Узел, Ссылка)`; снятие — `ПланыОбмена.УдалитьРегистрациюИзменений(...)`.

### Пример (регистрация изменения по узлу в обработчике)

```bsl
Процедура ПередЗаписью(Отказ)
    Если ОбменДанными.Загрузка Тогда
        Возврат;
    КонецЕсли;

    // Определить узлы, которым нужен этот объект (отбор по организации)
    МассивУзлов = ОбменДаннымиСервер.УзлыПоОрганизации(Организация);
    Для Каждого Узел Из МассивУзлов Цикл
        ПланыОбмена.ЗарегистрироватьИзменения(Узел, Ссылка);
    КонецЦикла;
КонецПроцедуры
```

### Пример (запись/чтение сообщения обмена)

```bsl
// Выгрузка изменений в файл
Узел = ПланыОбмена.ПланОбменаРИБ.НайтиПоКоду("ЦБ");
Запись = Новый ЗаписьXML;
Запись.ОткрытьФайл(ИмяФайла);
Выборка = ПланыОбмена.ВыбратьИзменения(Узел, ПланыОбмена.ПолучитьНомерОтправленного(Узел) + 1);
ПланыОбмена.ЗаписатьИзменения(Запись, Выборка);

// Загрузка изменений из файла
Чтение = Новый ЧтениеXML;
Чтение.ОткрытьФайл(ИмяФайла);
ПланыОбмена.ПрочитатьИзменения(Чтение, Узел);
```

## Обобщённая практика: XML-сериализация

### Средства платформы
- `XMLСтрока(Значение)` / `XMLЗначение(Тип, Строка)` — для примитивов и ссылок
- `СериализаторXDTO.ЗаписатьXML(Запись, Объект)` / `СериализаторXDTO.ПрочитатьXML(Чтение)` — для объектов/наборов записей
- `ЗначениеВСтрокуВнутр()` / `ЗначениеИзСтрокиВнутр()` — внутренний формат (только между базами одной конфигурации; не рекомендуется для интеграций)

### Правила
1. Для обмена между разными конфигурациями используйте канонический формат (EnterpriseData, #std771), а не внутренний формат.
2. XML-файлы — в UTF-8; имена файлов — латиницей (#std542 п.3.1).
3. При чтении XML всегда обрабатывайте `ОжидаемоеЗначение`/типы через `СериализаторXDTO` с указанием типа, иначе возможны уязвимости десериализации (#std770 — не выполнять произвольный код из XML).

### Пример (XDTO-сериализация объекта в XML)

```bsl
Запись = Новый ЗаписьXML;
Запись.УстановитьСтроку();
СериализаторXDTO.ЗаписатьXML(Запись, СсылкаНаОбъект);
XMLСтрока = Запись.Закрыть();

// Обратное чтение
Чтение = Новый ЧтениеXML;
Чтение.УстановитьСтроку(XMLСтрока);
Объект = СериализаторXDTO.ПрочитатьXML(Чтение);
```

## Обобщённая практика: Правила обмена и Конвертация данных

> Связано с #std637, #std771. Инструмент «Конвертация данных 2.0» (КД 2.0) и «Конвертация данных 3.0» (КД 3.0, EDT).

### Правила
1. Новые обмены между конфигурациями на БСП — на формате EnterpriseData (#std771), а не на новых ПКО КД 2.0.
2. ПКО (правила конвертации объектов): для классификаторов — поиск по идентификатору + полям поиска (#std637). Поля поиска — уникальные бизнес-ключи (БИК, ИНН, КПП).
3. Обработчики ПКО (`При загрузке`, `При выгрузке`, `Поля поиска`) — без обращения к связанным таблицам «через точку» (#std701 п.1.2).
4. В КД 3.0 (EDT) правила описываются декларативно; поддерживается версионирование формата.

## Обобщённая практика: Регистрация изменений и сообщения обмена

### Правила
1. Регистрация — на стороне источника, в транзакции записи объекта (#std701). Для ссылочных объектов при синхронизации через БСП регистрация отложенная (после загрузки всех данных сообщения).
2. Сообщения обмена нумеруются по узлу: `ПланыОбмена.ПолучитьНомерОтправленного(Узел)` / `ПланыОбмена.ПолучитьНомерПринятого(Узел)`. После успешной загрузки — `ПланыОбмена.УдалитьРегистрациюИзменений(Узел, НомерСообщения)`.
3. Обработка коллизий: при загрузке проверяйте `ОбменДанными.Загрузка` и не выполнять повторно бизнес-логику (#std773).

## #std542 (применимо к обмену): Доступ к файловой системе

**ITS:** https://its.1c.ru/db/v8std/content/542/hdoc | **v8std:** https://v8std.ru/std/542/

При файловом обмене (XML/JSON-файлы, ZIP-архивы сообщений):
1. Временные файлы сообщений — через `ПолучитьИмяВременногоФайла()` с удалением после обработки.
2. Кодировка UTF-8, имена файлов — латиницей (критично для Linux/macOS).
3. Учитывайте права пользователя сервера 1С при работе с каталогами обмена.

## Резюме ключевых принципов обмена данными

1. **Самодостаточность таблиц** в плане обмена — логика регистрации работает только с основной таблицей, без обращений «через точку» (#std701).
2. **EnterpriseData** — основной формат для обмена между конфигурациями на БСП; не использовать `AdditionalInfo` в типовых обменах (#std771).
3. **`ОбменДанными.Загрузка`** — обязательная проверка в `ПередЗаписью`/`ПриЗаписи`/`ПередУдалением`; объект грузится «как есть» (#std773).
4. **Классификаторы** — сопоставление по идентификатору + полям поиска; наборы записей (адресный) не участвуют в обмене (#std637).
5. **Безопасность/производительность** — таймауты (#std748), безопасный режим при `Выполнить` (#std770), UTF-8/латиница в файлах обмена (#std542).
