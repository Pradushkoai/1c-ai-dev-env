# Стандарты 1С: Организация работы конфигурации

**Источник:** [v8std.ru — Организация работы конфигурации](https://v8std.ru) | [ITS 1С](https://its.1c.ru/db/v8std/browse/13/-1/26/27)  
**Связанные инструменты:** `check_standards`, `analyze_bsl`, `solve_check`, `inspect`, `get_object_structure`

> Полный каталог стандартов 1С по организации работы конфигурации: имена объектов, часовые пояса, параметры сеанса, подсистемы, общие реквизиты, определяемые типы, пользовательские настройки, версии платформы, начальные действия, толстый клиент, разветвленная разработка, Linux/macOS. Стандарты #std550, #std643, #std413, #std543, #std677, #std704, #std557, #std785, #std556, #std680, #std709, #std723, #std706, #std731, #std759, #std798.

## #std550: Имена объектов метаданных в конфигурациях

**ITS:** https://its.1c.ru/db/v8std/content/550/hdoc | **v8std:** https://v8std.ru/std/550/

### Правила
1. Подсистемы именуйте по общим правилам: `Финансы`, `Маркетинг`, `НастройкаИАдминистирование`.
2. Параметры сеанса — `ТекущийПользователь`, `ОбменДаннымиВключен`.
3. Прикладные роли — по должности (`Бухгалтер`, `Кассир`); технические — по действию (`ДобавлениеИзменениеНСИ`).

## #std643: Работа в разных часовых поясах

**ITS:** https://its.1c.ru/db/v8std/content/643/hdoc | **v8std:** https://v8std.ru/std/643/

### Правила
1. В серверном контексте используйте `ТекущаяДатаСеанса` (а не `ТекущаяДата`), т.к. она приводит время к часовому поясу сеанса.
2. Для времени, независящего от сеанса (устаревание кеша, время фонового задания), применяйте `УниверсальноеВремя`.
3. Время сервера приводите к часовому поясу пользователя через `МестноеВремя`.

### Пример

```bsl
// ПРАВИЛЬНО: дата сеанса
ДатаНачала = ТекущаяДатаСеанса();

// Универсальное время для кеша
ДатаАктуальностиУниверсальная = УниверсальноеВремя(ПолнотекстовыйПоиск.ДатаАктуальности());

// Местное время пользователя
ДатаАктуальности = МестноеВремя(ДатаАктуальностиУниверсальная, ЧасовойПоясСеанса());
```

### Проверки
- EDT v8-code-style: `use-non-session-date`

## #std413: Использование параметров сеанса

**ITS:** https://its.1c.ru/db/v8std/content/413/hdoc | **v8std:** https://v8std.ru/std/413/

### Правила
1. Параметры сеанса инициализируйте в модуле сеанса «по требованию», а не при старте; используйте в запросах и RLS.
2. Не храните в них значения, нужные только клиенту — каждое обращение с клиента = серверный вызов.
3. Не кешируйте в параметрах вычисляемые значения — используйте общий модуль с повторным использованием возвращаемых значений.

### Пример

```bsl
// Модуль сеанса
Процедура УстановкаПараметровСеанса(ТребуемыеПараметры)
    Если ТребуемыеПараметры.Найти("ТекущийПользователь") <> Неопределено Тогда
        ПараметрыСеанса.ТекущийПользователь = Пользователи.ТекущийПользователь();
    КонецЕсли;
КонецПроцедуры
```

## #std543: Использование подсистем

**ITS:** https://its.1c.ru/db/v8std/content/543/hdoc | **v8std:** https://v8std.ru/std/543/

### Правила
1. Если интерфейсная и функциональная структура совпадают — одна иерархия подсистем с флажком «Включать в командный интерфейс».
2. При несовпадении создавайте отдельные «функциональные» подсистемы без флажка «Включать в командный интерфейс».
3. Объекты без визуального представления (общие модули, регламентные задания, подписки) включайте только в функциональные подсистемы.

## #std677: Использование общих реквизитов

**ITS:** https://its.1c.ru/db/v8std/content/677/hdoc | **v8std:** https://v8std.ru/std/677/

### Правила
1. Общие реквизиты применяйте для разделения данных (`Разделение данных = Разделять`) либо для служебного реквизита в нескольких объектах.
2. Не переносите в общие реквизиты прикладные поля вроде `Ответственный`, `Комментарий`.
3. Порядок реквизитов-разделителей задавайте по порядку установки связанных параметров сеанса.

## #std704: Использование определяемых типов

**ITS:** https://its.1c.ru/db/v8std/content/704/hdoc | **v8std:** https://v8std.ru/std/704/

### Правила
1. Определяемые типы используйте для данных, которые часто применяются и могут меняться при внедрении.
2. Для простого типа с прикладным смыслом — задаёт единую длину/точность во всех местах.
3. Для массового составного типа — фиксирует единый состав типов и упрощает доработку.

### Пример

```
Определяемый тип: НомерСчетаФактуры
Тип: Строка(50)
Используется: в документах счёт-фактура, книжке покупок, книжке продаж
```

## #std557: Работа с пользовательскими настройками

**ITS:** https://its.1c.ru/db/v8std/content/557/hdoc | **v8std:** https://v8std.ru/std/557/

### Правила
1. Персональные настройки храните в `ХранилищеОбщихНастроек`.
2. У пользователя должно быть право `СохранениеДанныхПользователя`.
3. Для каждой настройки — уникальный строковый ключ (`ОсновнаяОрганизация`, `ОсновнойСклад`).

### Пример

```bsl
// Сохранение
ХранилищеОбщихНастроек.Сохранить(
    "НастройкиПрограммы",
    "ЗадаватьВопросПриВыходе",
    ЗначениеНастройки);

// Загрузка
ЗначениеНастройки = ХранилищеОбщихНастроек.Загрузить(
    "НастройкиПрограммы",
    "ЗадаватьВопросПриВыходе");
```

## #std785: Версия платформы 1С:Предприятие для разработки

**ITS:** https://its.1c.ru/db/v8std/content/785/hdoc | **v8std:** https://v8std.ru/std/785/

### Правила
1. Минимальная версия платформы устанавливается не выше версии последнего сертифицированного релиза.
2. В исправительных релизах библиотек нельзя увеличивать минимальную версию платформы.
3. Рекомендуемая версия — последний опубликованный релиз, на котором ведутся разработка и тестирование.

## #std556: Начальные действия при работе конфигурации

**ITS:** https://its.1c.ru/db/v8std/content/556/hdoc | **v8std:** https://v8std.ru/std/556/

### Правила
1. Конфигурация должна автоматически определять первый запуск и первый запуск нового релиза, выполняя первичное заполнение и обновление данных.
2. Первичное заполнение разделяйте на обязательное и необязательное.
3. Неполная обработка должна контролироваться с предупреждением пользователю и протоколом в журнал регистрации.

### Пример

```bsl
Процедура ПриНачалеРаботыСистемы()
    Если Не ОбновлениеИнформационнойБазы.НеобходимоОбновление() Тогда
        Возврат;
    КонецЕсли;
    
    Если Не СтандартныеПодсистемыСервер.ПроверитьНеобходимостьЗавершенияОбновления() Тогда
        Возврат;
    КонецЕсли;
    
    ОбновлениеИнформационнойБазы.ВыполнитьОбновление();
КонецПроцедуры
```

## #std680: Поддержка толстого клиента

**ITS:** https://its.1c.ru/db/v8std/content/680/hdoc | **v8std:** https://v8std.ru/std/680/

### Правила
1. Модули менеджера, объекта, сеанса в управляемом приложении компилируются только на сервере.
2. Если нужно снять ограничение — выносите код обработчиков `ОбработкаПолученияПредставления` за инструкции препроцессора.
3. Для расстановки `#Если` используйте обработку с ИТС.

### Пример

```bsl
#Если Сервер Или ТолстыйКлиентОбычноеПриложение Или ВнешнееСоединение Тогда
    // Код, доступный в толстом клиенте
    ...
#Иначе
    ВызватьИсключение НСтр("ru = 'Недопустимый вызов объекта на клиенте.'");
#КонецЕсли
```

## #std709: Технология разветвленной разработки конфигураций

**ITS:** https://its.1c.ru/db/v8std/content/709/hdoc | **v8std:** https://v8std.ru/std/709/

### Правила
1. Для каждой исправительной версии создавайте новое хранилище на основе последней выпущенной версии (не копируйте основное).
2. В исправительной версии не должно быть объёмных доработок.
3. Плановая и исправительная версии различаются по срокам и составу изменений.

## #std723: Особенности разработки для Linux и macOS

**ITS:** https://its.1c.ru/db/v8std/content/723/hdoc | **v8std:** https://v8std.ru/std/723/

### Правила
1. Вместо COM (`COMОбъект`) используйте кроссплатформенные механизмы: `ras`/`rac`, `РабочийКаталогДанныхПользователя`.
2. Внешние компоненты разрабатывайте на технологии Native API (поддержка Linux, macOS, веб-клиента).
3. Для ключевых функций проверяйте `ТипПлатформы` и информируйте пользователя о недоступности.

### Пример

```bsl
Информация = Новый СистемнаяИнформация;
Если Информация.ТипПлатформы <> ТипПлатформы.Windows_x86
    И Информация.ТипПлатформы <> ТипПлатформы.Windows_x86_64 Тогда
    ПоказатьПредупреждение(,
        НСтр("ru = 'Печать в Microsoft Word доступна только при работе в Windows.'"));
КонецЕсли;
```

## #std706: Ограничения на переименование объектов метаданных

**ITS:** https://its.1c.ru/db/v8std/content/706/hdoc | **v8std:** https://v8std.ru/std/706/

### Правила
1. Не переименовывайте общий модуль с созданием нового с тем же именем — создайте новый модуль и перенесите код.
2. При удалении объекта с созданием нового с тем же именем сначала переименуйте устаревший с префиксом `Удалить`.
3. Ссылки на объекты метаданных храните в строковых реквизитах `Строка(255)` (полные имена).

## #std731: Требования к установке и обновлению

**ITS:** https://its.1c.ru/db/v8std/content/731/hdoc | **v8std:** https://v8std.ru/std/731/

### Правила
1. Дистрибутивы готовятся по главе 30 «Поставка и поддержка конфигурации» Руководства разработчика.
2. В каталоге версии — файл-манифест `1cv8.mft` с описанием шаблонов `.cf` и демонстрационных баз `.dt`.
3. В `Catalog` манифеста включайте наименование разработчика во избежание коллизий названий.

## #std759: Несущественные предупреждения проверки

**ITS:** https://its.1c.ru/db/v8std/content/759/hdoc | **v8std:** https://v8std.ru/std/759/

### Правила
1. Пустые обработчики оповещений и неразрешимые ссылки на метаданные обычно не критичны.
2. В регламентную проверку не включайте флажок «Поиск использования синхронных вызовов» — попадёт корректный серверный код.

**ITS:** https://its.1c.ru/db/v8std/content/798/hdoc | **v8std:** https://v8std.ru/std/798/
