# Антипаттерны BSL: модули и асинхронность

**Источник:** Практика разработки 1С + анализ ошибок компиляции | [v8std.ru](https://v8std.ru)  
**Связанные инструменты:** `check_1c_standards`, `architecture_analyzer`, `solve_check`, `analyze_bsl`

> Категория антипаттернов, выявленных при разработке клиентских общих модулей для мобильных приложений 1С. Все примеры — из реального кода, упавшего с ошибками компиляции/выполнения.

## Антипаттерн 1: `Ждать` без `Асинх` в сигнатуре функции

**ID правила:** `BSL-ASYNC-001`  
**Severity:** ERROR (код не скомпилируется)

### Симптом

При проверке модуля 1С выдаёт:

```
{ОбщийМодуль.Модуль1.Модуль(81,3)}: Оператор Ждать (Await) может употребляться
только в асинхронных процедурах или функциях
<<?>>Ждать УстановитьВнешнююКомпонентуАсинх(ИмяМакета);
```

### Причина

В 1С 8.3.18+ появилось ключевое слово `Асинх` (async) для функций, возвращающих обещание (Promise). Оператор `Ждать` (await) можно использовать **только** внутри функций, помеченных `Асинх`. Без этого маркера платформа не может гарантировать корректное приостановление выполнения.

### НЕПРАВИЛЬНО

```bsl
// Функция НЕ помечена как Асинх — но внутри есть Ждать → ошибка компиляции
Функция ПодключитьТСДАсинх(ИмяМакета, КлючПрофиля) Экспорт
    
    // ОШИБКА: Ждать без Асинх
    Ждать УстановитьВнешнююКомпонентуАсинх(ИмяМакета);
    Подключено = Ждать ПодключитьВнешнююКомпонентуАсинх(
        ИмяМакета, "InputDevice", ТипВнешнейКомпоненты.Native);
    
    Возврат Подключено;
    
КонецФункции
```

### ПРАВИЛЬНО

```bsl
// Функция помечена как Асинх — Ждать разрешён
Асинх Функция ПодключитьТСДАсинх(ИмяМакета, КлючПрофиля) Экспорт
    
    Ждать УстановитьВнешнююКомпонентуАсинх(ИмяМакета);
    Подключено = Ждать ПодключитьВнешнююКомпонентуАсинх(
        ИмяМакета, "InputDevice", ТипВнешнейКомпоненты.Native);
    
    Возврат Подключено;
    
КонецФункции
```

### Когда `Ждать` разрешено без `Асинх`

Только в **формах** в обработчиках, у которых платформа неявно добавляет асинхронность:

```bsl
&НаКлиенте
Процедура ПриОткрытии(Отказ)
    // Здесь Ждать разрешено — платформа сама делает обработчик асинхронным
    Результат = Ждать ПодключитьТСДАсинх("Макет", "MERCHECH_S40");
КонецПроцедуры
```

Но даже в формах лучше явно вызывать асинхронные функции через `ОписаниеОповещения`, если сигнатура обработчика не async.

### Связанные стандарты

- **#std629** — Минимизация кода на клиенте (асинхронные вызовы — на сервере)
- **#std755** — Длительные операции на клиенте (использовать асинхронность)

### Методы платформы, возвращающие обещания (требуют `Ждать`)

| Метод | Назначение | Контекст |
|-------|------------|----------|
| `УстановитьВнешнююКомпонентуАсинх(ИмяМакета)` | Установка ВК | **только клиент** |
| `ПодключитьВнешнююКомпонентуАсинх(...)` | Подключение ВК | **только клиент** |
| `ПоказатьВопросАсинх(...)` | Асинхронный диалог | **только клиент** |
| `ПоказатьПредупреждениеАсинх(...)` | Асинхронное предупреждение | **только клиент** |
| `ПоказатьВводСтрокиАсинх(...)` | Асинхронный ввод | **только клиент** |
| `ПоказатьВводЧислаАсинх(...)` | Асинхронный ввод числа | **только клиент** |
| `ПоказатьВводДатыАсинх(...)` | Асинхронный ввод даты | **только клиент** |
| `ОткрытьФормуАсинх(...)` | Асинхронное открытие формы | **только клиент** |
| `HTTPСоединение.ПолучитьАсинх(...)` (8.3.21+) | Асинхронный HTTP GET | **только клиент** |
| `HTTPСоединение.ОтправитьДляОбработкиАсинх(...)` (8.3.21+) | Асинхронный HTTP POST | **только клиент** |
| `КопироватьФайлАсинх(...)` | Асинхронное копирование файла | **только клиент** |
| `НайтиФайлыАсинх(...)` | Асинхронный поиск файлов | **только клиент** |
| `ЗаписатьАсинх(...)` | Асинхронная запись | **только клиент** |

Любой из этих методов внутри общего модуля требует, чтобы:
1. Функция была `Асинх`
2. Модуль имел клиентские флаги (НЕ серверные!)

## Антипаттерн 1.1: `Асинх` в серверном модуле (или «Мобильное приложение-сервер»)

**ID правила:** `BSL-ASYNC-003`  
**Severity:** ERROR (код не скомпилируется)

> ⚠️ **Это критически важное правило, подтверждённое официальной документацией 1С и практикой.**

### Симптом

При проверке общего модуля 1С выдаёт:

```
{ОбщийМодуль.Модуль1.Модуль(73,1)}: Недопустимое объявление асинхронной процедуры или функции
<<?>>Асинх Функция ПодключитьТСДАсинх(...) Экспорт (Проверка: Сервер)

{ОбщийМодуль.Модуль1.Модуль(73,1)}: Недопустимое объявление асинхронной процедуры или функции
<<?>>Асинх Функция ПодключитьТСДАсинх(...) Экспорт (Проверка: Мобильное приложение-сервер)
```

### Причина

Ключевые слова `Асинх Процедура` и `Асинх Функция` в платформе 1С:Предприятие **применимы только в клиентском контексте**. На сервере (включая «Мобильное приложение-сервер») их использовать **нельзя**.

**Техническое обоснование:**

1. Тип данных `Обещание` (Promise) доступен **только в клиентском контексте** (тонкий, толстый, мобильный, веб-клиент)
2. Асинхронные методы (`…Асинх`) создавались платформой **исключительно для клиентских контекстов**, чтобы не блокировать UI
3. На сервере блокировать нечего — сеанс изолирован, поэтому асинхронность не нужна
4. Для распараллеливания серверных операций используется **фоновое задание** (`ФоновыеЗадания.Выполнить`), а не `Асинх`/`Ждать`

### Где можно и нельзя использовать `Асинх` (полная таблица)

| Модуль / Контекст компиляции | `Асинх` поддерживается? | Пояснение |
|---|---|---|
| **Модуль формы**, директива `&НаКлиенте` | ✅ **Да** | Основное место применения |
| **Модуль формы**, `&НаСервере` / `&НаСервереБезКонтекста` | ❌ **Нет** | Ошибка компиляции |
| **Общий модуль**, флаг «Клиент (управляемое приложение)» | ✅ **Да** | Только клиентский контекст |
| **Общий модуль**, флаг «Сервер» | ❌ **Нет** | Серверный контекст |
| **Общий модуль**, флаг «Мобильное приложение (клиент)» | ✅ **Да** | Клиент мобильного приложения |
| **Общий модуль**, флаг «Мобильное приложение (сервер)» | ❌ **Нет** | Серверная часть мобильного приложения |
| **Модуль объекта** (справочник, документ, …) | ❌ **Нет** | Только серверный контекст |
| **Модуль менеджера** объекта | ❌ **Нет** | Только серверный контекст |
| **Модуль команды**, обработчик `&НаКлиенте` | ✅ **Да** | Обработчики команд формы на клиенте |
| **Модуль команды**, область `&НаСервере` | ❌ **Нет** | — |
| **Веб-клиент**, `&НаКлиенте` | ✅ **Да** | Асинхронность особенно важна для браузеров |

### НЕПРАВИЛЬНО

```bsl
// Общий модуль «ОбщийМодуль1»
// Флаги: Сервер + Мобильное приложение (сервер)
// ❌ ОШИБКА: Асинх в серверном модуле

Асинх Функция ПодключитьТСДАсинх(ИмяМакета, КлючПрофиля) Экспорт
    
    Ждать УстановитьВнешнююКомпонентуАсинх(ИмяМакета);
    Подключено = Ждать ПодключитьВнешнююКомпонентуАсинх(ИмяМакета, "InputDevice", ТипВнешнейКомпоненты.Native);
    
    Возврат Подключено;
    
КонецФункции
```

### ПРАВИЛЬНО: разделить на клиентский и серверный модули

**Архитектурное правило:** не пытаться писать «универсальный» асинхронный код в одном общем модуле с флагом «Сервер». Создаётся **минимум два** общих модуля:

| Общий модуль | Флаги | Что внутри |
|---|---|---|
| `ИнтеграцияТСДКлиент` | Клиент (управляемое приложение) + Мобильное приложение (клиент) | `Асинх Функция ПодключитьТСДАсинх(...)` — клиентский API с `Ждать` |
| `ИнтеграцияТСДСервер` | Сервер + Мобильное приложение (сервер) | Синхронные серверные функции (если нужны) |
| `ИнтеграцияТСДВызовСервера` | Сервер + Клиент (для вызова с клиента) | Тонкий прокси-модуль для `&НаСервереБезКонтекста` |

**Клиентский модуль (с `Асинх`):**

```bsl
// Общий модуль «ИнтеграцияТСДКлиент»
// Флаги: ✅ Клиент (управляемое приложение) + ✅ Мобильное приложение (клиент)
// Флаги: ❌ Сервер — СНЯТЬ, ❌ Мобильное приложение (сервер) — СНЯТЬ

#Область ПрограммныйИнтерфейс

Асинх Функция ПодключитьТСДАсинх(ИмяМакета, КлючПрофиля) Экспорт
    
    Результат = Новый Структура("Успех, Ошибка, Драйвер", Ложь, "", Неопределено);
    
    // 1. Получить профиль с сервера (синхронный серверный вызов)
    Профиль = ИнтеграцияТСДВызовСервера.ПолучитьПрофильТСД(КлючПрофиля);
    
    // 2. Асинхронно установить и подключить компоненту (только на клиенте!)
    Попытка
        Ждать УстановитьВнешнююКомпонентуАсинх(ИмяМакета);
        Подключено = Ждать ПодключитьВнешнююКомпонентуАсинх(
            ИмяМакета, "InputDevice", ТипВнешнейКомпоненты.Native);
        
        Если Не Подключено Тогда
            Результат.Ошибка = "Не удалось подключить компоненту";
            Возврат Результат;
        КонецЕсли;
        
        Драйвер = Новый("AddIn.InputDevice.InputDevice");
        // ... настройка параметров ...
        Драйвер.Подключить(Профиль.BluetoothDevice);
        
        Результат.Успех = Истина;
        Результат.Драйвер = Драйвер;
    Исключение
        Результат.Ошибка = ОписаниеОшибки();
    КонецПопытки;
    
    Возврат Результат;
    
КонецФункции

#КонецОбласти
```

**Серверный модуль (синхронный, без `Асинх`):**

```bsl
// Общий модуль «ИнтеграцияТСДСервер»
// Флаги: ✅ Сервер + ✅ Мобильное приложение (сервер)
// Флаги: ❌ Клиент — СНЯТЬ

#Область ПрограммныйИнтерфейс

// Только синхронные функции!
Функция ПолучитьПрофильТСД(КлючПрофиля) Экспорт
    // Чтение из регистра/справочника — синхронно
    Возврат РегистрыСведений.ПрофилиТСД.Получить(
        Новый Структура("Ключ", КлючПрофиля));
КонецФункции

// Если на сервере нужно отправить HTTP-запрос к ТСД — ТОЛЬКО СИНХРОННО
Функция ОтправитьКомандуНаТСД(Адрес, Порт, Команда) Экспорт
    Попытка
        Соединение = Новый HTTPСоединение(Адрес, Порт);
        Запрос = Новый HTTPЗапрос("/" + Команда);
        Ответ = Соединение.Получить(Запрос);  // ✅ синхронно, без Ждать
        Возврат Ответ.КодСостояния = 200;
    Исключение
        Возврат Ложь;
    КонецПопытки;
КонецФункции

#КонецОбласти
```

### Версии платформы

| Версия | Что появилось |
|---|---|
| **8.3.18** (8.3.18.891/.1128) | 🎯 Появились `Асинх`, `Ждать`, тип `Обещание`. Только в клиентском контексте. Режим совместимости ≥ 8.3.18 |
| **8.3.21** | Асинхронные HTTP-методы на клиенте (`HTTPСоединение.…Асинх`). На сервере HTTP — только синхронно |

> ⚠️ Если режим совместимости конфигурации < 8.3.18, ключевые слова `Асинх` и `Ждать` не распознаются платформой вообще.

### Альтернативы для серверных длительных операций

На сервере `Асинх`/`Ждать` недоступны. Используется:

1. **Фоновое задание** (`ФоновыеЗадания.Выполнить`) + опрос на клиенте через `ПодключитьОбработчикОжидания`
2. **БСП «ДлительныеОперации»** — `ДлительныеОперации.ВыполнитьВФоновомЗадании` + `ДлительныеОперацииКлиент.ОжидатьЗавершение`
3. **Синхронные методы** — если операция не очень длительная (< 5 секунд)

См. стандарт 1С [#std642](https://its.1c.ru/db/v8std/content/642/hdoc) «Длительные операции на сервере».

### Связанные стандарты

- **#std629** — Минимизация кода на клиенте
- **#std642** — Длительные операции на сервере (через фоновые задания)
- **#std755** — Длительные операции на клиенте

## Антипаттерн 2: `Перем ... Экспорт` в общем модуле

**ID правила:** `BSL-MODULE-VAR-001`  
**Severity:** ERROR (код не скомпилируется)

### Симптом

При проверке модуля 1С выдаёт:

```
{ОбщийМодуль.Модуль1.Модуль(855,1)}: Данный модуль может содержать только
определения процедур и функций
<<?>>Перем ДрайверТСД Экспорт;
```

### Причина

В общих модулях 1С **запрещено** объявлять экспортные переменные. Это ограничение платформы — общий модуль должен содержать только определения процедур и функций. Кроме того, переменные модуля в общих модулях — плохая практика (#std639), потому что:

1. В клиент-серверной архитектуре у каждого клиента свой экземпляр модуля
2. В мобильном приложении переменные теряются при сворачивании/восстановлении
3. Состояние модуля сложно контролировать и отлаживать
4. Это нарушает принцип stateless-дизайна общих модулей

### НЕПРАВИЛЬНО

```bsl
// В общем модуле — НЕЛЬЗЯ
Перем ДрайверТСД Экспорт;       // ОШИБКА КОМПИЛЯЦИИ
Перем АктивныйПрофиль Экспорт;  // ОШИБКА КОМПИЛЯЦИИ

Функция ПодключитьТСД(ИмяМакета) Экспорт
    ДрайверТСД = Новый("AddIn.InputDevice.InputDevice");
    Возврат Истина;
КонецФункции

Функция ПолучитьДрайвер() Экспорт
    Возврат ДрайверТСД;  // никогда не сработает корректно
КонецФункции
```

### ПРАВИЛЬНО (вариант 1): возвращать драйвер из функции

```bsl
// Общий модуль — без состояния, возвращает результат
Асинх Функция ПодключитьТСДАсинх(ИмяМакета, КлючПрофиля) Экспорт
    
    Результат = Новый Структура("Успех, Ошибка, Драйвер", Ложь, "", Неопределено);
    
    // ... подключение ...
    Результат.Драйвер = Новый("AddIn.InputDevice.InputDevice");
    Результат.Успех = Истина;
    
    Возврат Результат;  // драйвер возвращается вызывающему
    
КонецФункции
```

Форма хранит ссылку:

```bsl
// В модуле формы
&НаКлиенте
Перем ДрайверТСД;  // переменная ФОРМЫ — можно

&НаКлиенте
Асинх Процедура ПриОткрытии(Отказ)
    Результат = Ждать ИнтеграцияТСДКлиент.ПодключитьТСДАсинх("Макет", "MERCHECH_S40");
    Если Результат.Успех Тогда
        ДрайверТСД = Результат.Драйвер;  // сохраняем в форме
    КонецЕсли;
КонецПроцедуры
```

### ПРАВИЛЬНО (вариант 2): хранить в параметре сеанса

Если состояние должно переживать перезапуск формы:

```bsl
// Серверный модуль
Процедура УстановитьДрайверТСД(Драйвер) Экспорт
    // Сохранить в параметр сеанса (только для примитивных типов)
    // Для объектов AddIn — не подходит, нужно использовать форму
КонецПроцедуры
```

### ПРАВИЛЬНО (вариант 3): модуль объекта / модуль формы

Переменные модуля разрешены в:
- ✅ Модулях форм (`&НаКлиенте Перем ...`)
- ✅ Модулях объектов (СправочникОбъект, ДокументОбъект)
- ✅ Модулях менеджеров (с осторожностью)
- ❌ Общих модулях — `Перем ... Экспорт` запрещена
- ⚠️ Общих модулях — `Перем ...` без Экспорт разрешена, но не рекомендуется (#std639)

### Связанные стандарты

- **#std639** — Использование переменных в программных модулях (вместо переменных модуля — другие средства)
- **#std508** — Использование переменных (только локальные)
- **#std580** — Минимизация кода на клиенте

### Связанное правило analyzer'а

У нас уже реализовано в `check_1c_standards`:
- `no-export-variables` — детектирует `Перем ... Экспорт` (#std639)

Но нужно усилить: `Перем ... Экспорт` в общем модуле — это **ERROR компиляции**, а не просто warning. Стоит добавить отдельное правило `no-module-export-var-in-common-module` с severity=error.

## Антипаттерн 3: Хранение состояния в общем модуле (без `Экспорт`)

**ID правила:** `BSL-MODULE-VAR-002`  
**Severity:** WARNING (код скомпилируется, но поведение некорректное)

### Симптом

Код работает локально у разработчика, но в production:
- У одного пользователя состояние видно другому
- При перезапуске формы состояние теряется
- При переключении между клиентами (тонкий/веб/мобильный) состояние неконсистентно

### Причина

Даже без `Экспорт`, переменные модуля в общих модулях 1С имеют непредсказуемое время жизни:

- В **толстом клиенте** — переменные живут до закрытия приложения
- В **тонком клиенте** — у каждого клиентского сеанса свой экземпляр
- В **веб-клиенте** — переинициализация при перезагрузке страницы
- В **мобильном приложении** — переинициализация при сворачивании
- На **сервере** — разделяются между сеансами (опасно!)

### НЕПРАВИЛЬНО

```bsl
// Общий модуль — состояние в переменной модуля
Перем КэшДанных;

Функция ПолучитьДанные(Ключ) Экспорт
    Если КэшДанных = Неопределено Тогда
        КэшДанных = Новый Соответствие;
    КонецЕсли;
    
    Если Не КэшДанных.Свойство(Ключ) Тогда
        КэшДанных.Вставить(Ключ, ЗагрузитьДанные(Ключ));
    КонецЕсли;
    
    Возврат КэшДанных[Ключ];
КонецФункции
```

### ПРАВИЛЬНО

Использовать один из механизмов платформы:

```bsl
// Вариант 1: Общий модуль с повторным использованием возвращаемых значений
// (флаг "Повторное использование возвращаемых значений" = "На время вызова)
Функция ПолучитьДанные(Ключ) Экспорт
    Возврат ЗагрузитьДанные(Ключ);  // платформа сама кэширует
КонецФункции

// Вариант 2: Регистр сведений для серверного кэша
Функция ПолучитьДанные(Ключ) Экспорт
    Возврат РегистрыСведений.КэшДанных.Получить(Новый Структура("Ключ", Ключ));
КонецФункции

// Вариант 3: Хранилище общих настроек (для клиентских настроек)
Функция ПолучитьНастройку(Ключ) Экспорт
    Возврат ХранилищеОбщихНастроек.Загрузить("МоёПриложение", Ключ);
КонецФункции
```

### Связанные стандарты

- **#std639** — Не использовать переменные модуля
- **#std557** — ХранилищеОбщихНастроек для пользовательских настроек
- **#std724** — Общие модули с повторным использованием возвращаемых значений

## Антипаттерн 4: Смешивание `Асинх` и не-async обработчиков в форме

**ID правила:** `BSL-ASYNC-002`  
**Severity:** WARNING

### Симптом

Код компилируется, но `Ждать` в обработчике формы не работает как ожидается — UI не обновляется, данные не успевают сохраниться.

### Причина

Обработчики форм 1С (`ПриОткрытии`, `ПриИзменении`, и т.д.) — **синхронные**. Если внутри них вызвать `Ждать`, платформа приостановит обработчик, но UI заблокируется. Правильно — использовать `ОписаниеОповещения` для отложенной обработки.

### НЕПРАВИЛЬНО

```bsl
&НаКлиенте
Процедура НоменклатураПриИзменении(Элемент)
    // Ждать блокирует UI
    ДанныеНоменклатуры = Ждать ЗагрузитьДанныеНоменклатурыАсинх(Объект.Номенклатура);
    
    // Это выполнится только после возврата из Ждать
    Объект.Цена = ДанныеНоменклатуры.Цена;
КонецПроцедуры
```

### ПРАВИЛЬНО

```bsl
&НаКлиенте
Процедура НоменклатураПриИзменении(Элемент)
    // Запускаем асинхронно, UI не блокируется
    Оповещение = Новый ОписаниеОповещения("ПослеЗагрузкиДанных", ЭтотОбъект);
    НачатьЗагрузкуДанныхНоменклатуры(Оповещение, Объект.Номенклатура);
КонецПроцедуры

&НаКлиенте
Процедура ПослеЗагрузкиДанных(ДанныеНоменклатуры, ДопПараметры) Экспорт
    Если ДанныеНоменклатуры <> Неопределено Тогда
        Объект.Цена = ДанныеНоменклатуры.Цена;
    КонецЕсли;
КонецПроцедуры
```

### Связанные стандарты

- **#std400** — Информирование пользователя (асинхронные методы)
- **#std755** — Длительные операции на клиенте

## Правила для генератора кода

При генерации BSL-кода ИИ-агентом (через наш `code_generator` или `bsl_templates`) нужно следовать этим правилам:

1. **Если в функции используется `Ждать`** → обязательно `Асинх Функция ...` в сигнатуре
2. **Если в общем модуле** → никаких `Перем ... Экспорт` (вообще без переменных модуля)
3. **Если нужно состояние** → возвращать через результат функции, хранить в форме
4. **Если обработчик формы** → использовать `ОписаниеОповещения`, а не `Ждать`

### Шаблон для асинхронной функции общего модуля

```bsl
// Заголовок: обязательно Асинх
Асинх Функция МойМетодАсинх(Параметр1, Параметр2 = Неопределено) Экспорт
    
    Результат = Новый Структура("Успех, Ошибка, Данные", Ложь, "", Неопределено);
    
    Попытка
        // ... асинхронные операции с Ждать ...
        Ждать ЧтоТоАсинх(Параметр1);
        
        Результат.Успех = Истина;
        Результат.Данные = ПолученныеДанные;
    Исключение
        Результат.Ошибка = КраткоеПредставлениеОшибки(ИнформацияОбОшибке());
    КонецПопытки;
    
    Возврат Результат;  // всегда возвращаем структуру
    
КонецФункции
```

### Шаблон для общего модуля без состояния

```bsl
// Общий модуль — только функции, без переменных
#Область ПрограммныйИнтерфейс

Функция Метод1(Параметр) Экспорт
    // ... вычисление ...
    Возврат Результат;
КонецФункции

Функция Метод2(Параметр) Экспорт
    // ... вычисление ...
    Возврат Результат;
КонецФункции

#КонецОбласти

#Область СлужебныеПроцедурыИФункции

// Локальные функции — без Экспорт
Функция Вспомогательная(Параметр)
    Возврат Параметр;
КонецФункции

#КонецОбласти

// НИКАКИХ Перем ... в конце модуля
```

## Чек-лист для reviewer'а

При review BSL-кода общего модуля проверить:

- [ ] Все функции с `Ждать` помечены `Асинх`
- [ ] **`Асинх` функции находятся только в клиентских модулях** (не в серверных!)
- [ ] **Общий модуль с `Асинх` НЕ имеет флагов «Сервер» и «Мобильное приложение (сервер)»**
- [ ] Нет `Перем ... Экспорт` в общем модуле
- [ ] Нет переменных модуля вообще (даже без `Экспорт`)
- [ ] Состояние возвращается через результат функции
- [ ] Обработчики форм используют `ОписаниеОповещения` или `Асинх`, а не синхронный `Ждать` в синхронном обработчике
- [ ] Структура результата единообразна: `{Успех, Ошибка, Данные}`
- [ ] Режим совместимости конфигурации ≥ 8.3.18 (иначе `Асинх`/`Ждать` не распознаются)
- [ ] Серверные HTTP-запросы — только синхронные (без `…Асинх`)
- [ ] Для длительных серверных операций используются **фоновые задания** (#std642)
