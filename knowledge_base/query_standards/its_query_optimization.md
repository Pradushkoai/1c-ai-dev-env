## #std729: Общие требования по разработке оптимальных запросов

**ITS:** https://its.1c.ru/db/v8std/content/729/hdoc

### Суть
Общие принципы написания эффективных запросов 1С.

### Правила
- Не выбирать лишние поля (`ВЫБРАТЬ *`)
- Использовать параметры (`&Параметр`), а не конкатенацию
- Ограничивать выборку через `ГДЕ` и `ПЕРВЫЕ N`
- Использовать индексы (поля индекса в начале условия `ГДЕ`)
- Избегать функций в `ГДЕ` — блокируют индекс
- Использовать `ВЫБРАТЬ РАЗЛИЧНЫЕ` только когда нужно

### Связанные стандарты
- Все нижеследующие стандарты — конкретизация этого

## #std652: Несоответствие индексов и условий запроса

**ITS:** https://its.1c.ru/db/v8std/content/652/hdoc

### Суть
Поля в условии `ГДЕ` должны соответствовать индексам таблицы — иначе full scan.

### Правила
1. Индекс должен содержать **все поля** условия
2. Поля должны быть **в начале** индекса
3. Поля должны быть **последовательными** (без пропусков)
4. Не использовать функции на полях индекса в `ГДЕ`

### Пример

```bsl
// ПРАВИЛЬНО: поля соответствуют индексу (Номенклатура, Склад)
"ГДЕ Т.Номенклатура = &Ном И Т.Склад = &Скл"

// ПЛОХО: функция блокирует индекс
"ГДЕ НАЧАЛОПЕРИОДА(Т.Дата, ДЕНЬ) = &Дата"
// Лучше: ГДЕ Т.Дата МЕЖДУ &ДатаНачала И &ДатаКонца
```

## #std654: Разыменование ссылочных полей составного типа

**ITS:** https://its.1c.ru/db/v8std/content/654/hdoc

### Суть
Обращение через точку к составному типу (`Регистратор.Дата`) выполняет запрос по ВСЕМ типам.

### Правила
- Использовать `ВЫРАЗИТЬ(... КАК Документ.Имя)` для конкретного типа
- Избегать глубокого разыменования (`Т.Поле.Подполе.Подполе`)
- Чрезмерное использование `.Ссылка` — скрытые JOIN

### Пример

```bsl
// ПЛОХО: запрос по всем типам регистратора
"ВЫБРАТЬ Т.Регистратор.Дата"

// ПРАВИЛЬНО: конкретный тип
"ВЫБРАТЬ ВЫРАЗИТЬ(Т.Регистратор КАК Документ.Поступление).Дата"
```

### Связанные диагностики
- BSL LS: `RefOveruse`

## #std655: Ограничения на соединения с вложенными запросами и виртуальными таблицами

**ITS:** https://its.1c.ru/db/v8std/content/655/hdoc

### Суть
JOIN с подзапросами и виртуальными таблицами — крайне медленно.

### Правила
- НЕ JOIN'ить с подзапросами — использовать временные таблицы
- НЕ JOIN'ить с виртуальными таблицами напрямую — сначала во временную
- Виртуальную таблицу поместить во временную с `ИНДЕКСИРОВАТЬ ПО`

### Пример

```bsl
// ПЛОХО: JOIN с подзапросом
"... ЛЕВОЕ СОЕДИНЕНИЕ (ВЫБРАТЬ ... СГРУППИРОВАТЬ ПО ...) КАК Итоги ..."

// ПРАВИЛЬНО: временная таблица + JOIN
"ВЫБРАТЬ ... ПОМЕСТИТЬ ВТИтоги ... СГРУППИРОВАТЬ ПО ... ИНДЕКСИРОВАТЬ ПО Ключ;
|ВЫБРАТЬ ... ИЗ ... ЛЕВОЕ СОЕДИНЕНИЕ ВТИтоги ПО ..."
```

### Связанные диагностики
- BSL LS: `JoinWithSubQuery`, `JoinWithVirtualTable`
- Наши: O004

## #std656: Ограничения на вложенные запросы в условии соединения

**ITS:** https://its.1c.ru/db/v8std/content/656/hdoc

### Суть
Вложенные запросы в условии `ПО` (ON) — медленно.

### Правила
- Не использовать подзапросы в `ПО`
- Заменять на временную таблицу + JOIN по простому полю

### Пример

```bsl
// ПЛОХО: подзапрос в ПО
"... ЛЕВОЕ СОЕДИНЕНИЕ Б ПО А.Ключ = (ВЫБРАТЬ МАКСИМУМ(Б.Ключ) ИЗ ...)"

// ПРАВИЛЬНО: временная таблица
"ВЫБРАТЬ МАКСИМУМ(Б.Ключ) КАК Ключ ПОМЕСТИТЬ ВТКлючи ...;
|ВЫБРАТЬ ... ЛЕВОЕ СОЕДИНЕНИЕ ВТКлючи ПО А.Ключ = ВТКлючи.Ключ"
```

## #std657: Обращения к виртуальным таблицам

**ITS:** https://its.1c.ru/db/v8std/content/657/hdoc

### Суть
Виртуальные таблицы (Остатки, Обороты, СрезПоследних) — должны вызываться с параметрами-фильтрами.

### Правила
- Фильтр должен быть **в параметрах** виртуальной таблицы, а не в `ГДЕ`
- Параметр `Граница периода` — обязательный для Остатки/Обороты
- Не вызывать виртуальную таблицу без фильтра по измерениям
- Помещать результат во временную таблицу при JOIN

### Пример

```bsl
// ПРАВИЛЬНО: фильтр в параметрах
"ИЗ РегистрНакопления.Товары.Остатки(&Период, Номенклатура В (&Список)) КАК Остатки"

// ПЛОХО: фильтр в WHERE
"ИЗ РегистрНакопления.Товары.Остатки(&Период) КАК Остатки
|ГДЕ Остатки.Номенклатура В (&Список)"
```

### Связанные диагностики
- BSL LS: `VirtualTableCallWithoutParameters`
- Наши: O005

## #std658: Эффективные условия запросов

**ITS:** https://its.1c.ru/db/v8std/content/658/hdoc

### Суть
Условия в `ГДЕ` должны быть эффективными — использовать индексы, избегать `ИЛИ`.

### Правила
- Избегать `ИЛИ` в `ГДЕ` — блокирует индексы, заменить на `ОБЪЕДИНИТЬ ВСЕ`
- Не использовать `НЕ` перед полем — блокирует индекс
- Не использовать функции на полях индекса
- Использовать `В (&Массив)` вместо последовательности `ИЛИ`
- `ЕСТЬ NULL` / `НЕ ЕСТЬ NULL` — эффективно

### Пример

```bsl
// ПЛОХО: ИЛИ блокирует индекс
"ГДЕ Т.Артикул = &Арт ИЛИ Т.Код = &Код"

// ПРАВИЛЬНО: ОБЪЕДИНИТЬ ВСЕ
"ВЫБРАТЬ ... ГДЕ Т.Артикул = &Арт
|ОБЪЕДИНИТЬ ВСЕ
|ВЫБРАТЬ ... ГДЕ Т.Код = &Код"

// ПЛОХО: НЕ блокирует индекс
"ГДЕ НЕ Т.ПометкаУдаления = ИСТИНА"

// ПРАВИЛЬНО
"ГДЕ Т.ПометкаУдаления = ЛОЖЬ"
```

### Связанные диагностики
- BSL LS: `LogicalOrInTheWhereSectionOfQuery`
- Наши: O002

## #std733: Эффективное обращение к виртуальной таблице «Остатки»

**ITS:** https://its.1c.ru/db/v8std/content/733/hdoc

### Суть
Специфика виртуальной таблицы Остатки — правильное использование параметров.

### Правила
- Передавать фильтр по измерениям в параметрах
- Указывать дату — не использовать пустую
- При JOIN — сначала во временную таблицу с индексом
- Использовать `Граница периода` для оптимизации

### Пример

```bsl
// ПРАВИЛЬНО: фильтр + индекс во временной таблице
"ВЫБРАТЬ Остатки.Номенклатура, Остатки.КоличествоОстаток
|ПОМЕСТИТЬ ВТОстатки
|ИЗ РегистрНакопления.Товары.Остатки(&Период, Номенклатура В (&Список))
|ИНДЕКСИРОВАТЬ ПО Номенклатура;
|ВЫБРАТЬ ... ИЗ Справочник.Номенклатура ЛЕВОЕ СОЕДИНЕНИЕ ВТОстатки ..."
```

## #std777: Использование временных таблиц

**ITS:** https://its.1c.ru/db/v8std/content/777/hdoc

### Суть
Временные таблицы — для оптимизации сложных запросов и JOIN.

### Правила
- `ПОМЕСТИТЬ` — создать временную таблицу
- `ИНДЕКСИРОВАТЬ ПО` — создать индекс для JOIN
- `УНИЧТОЖИТЬ` — удалить когда не нужна (экономит память)
- Использовать вместо подзапросов при JOIN
- Название с префиксом `ВТ` (Временная Таблица)

### Пример

```bsl
// ПРАВИЛЬНО: временная таблица с индексом
"ВЫБРАТЬ ... ПОМЕСТИТЬ ВТДанные
|ИЗ ...
|ГДЕ ...
|ИНДЕКСИРОВАТЬ ПО Ключ;
|ВЫБРАТЬ ... ИЗ ВТДанные ЛЕВОЕ СОЕДИНЕНИЕ ..."

// ПЛОХО: временная таблица без индекса
"ВЫБРАТЬ ... ПОМЕСТИТЬ ВТДанные ИЗ ...;
|ВЫБРАТЬ ... ИЗ ВТДанные ЛЕВОЕ СОЕДИНЕНИЕ ..."
```

### Связанные диагностики
- Наши: O007

## #std791: Дополнительные индексы

**ITS:** https://its.1c.ru/db/v8std/content/791/hdoc

### Суть
Дополнительные индексы для полей, по которым часто ищут, но которые не в основном индексе.

### Правила
- Создавать индексы для полей, часто используемых в `ГДЕ`
- Не создавать избыточные индексы — замедляют запись
- Индекс должен содержать поля условия в порядке убывания селективности
- Поля индекса должны быть последовательными
