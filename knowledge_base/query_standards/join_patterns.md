# Паттерны JOIN в запросах 1С

**Источник:** [ITS #std655](https://its.1c.ru/db/v8std/content/655/hdoc), [ITS #std656](https://its.1c.ru/db/v8std/content/656/hdoc), [ITS #std435](https://its.1c.ru/db/v8std/content/435/hdoc), [v8std.ru](https://v8std.ru)  
**Связанные инструменты:** `generate_query`, `analyze_queries`, `validate_query_static`, `solve_check`, `dependency_graph.find_join_path`

> Глубокий справочник по всем видам соединений в 1С: когда применять, как оптимизировать, какие ловушки подстерегают. Подчиняется #std655, #std656, #std435, #std654.

---

## 1. Виды соединений в 1С

| Соединение | Что возвращает | Когда применять |
|------------|----------------|----------------|
| `ВНУТРЕННЕЕ СОЕДИНЕНИЕ` (INNER) | Только строки с совпадением в обеих таблицах | Когда нужны обе стороны связи |
| `ЛЕВОЕ СОЕДИНЕНИЕ` (LEFT OUTER) | Все строки левой таблицы + совпавшие правые | Когда нужны все строки основной таблицы |
| `ПРАВОЕ СОЕДИНЕНИЕ` (RIGHT OUTER) | Все строки правой + совпавшие левые | Редко — обычно заменяется на LEFT с перестановкой |
| `ПОЛНОЕ СОЕДИНЕНИЕ` (FULL OUTER) | Все строки обеих таблиц | **Избегать** — см. #std435 |

## 2. Стандартные паттерны

### 2.1. ЛЕВОЕ СОЕДИНЕНИЕ с проверкой IS NULL (анти-join)

**Когда:** нужно найти строки левой таблицы, для которых нет соответствия в правой.

```bsl
// ПРАВИЛЬНО: найти номенклатуру, которая ни разу не продавалась
ВЫБРАТЬ
    Ном.Ссылка КАК Номенклатура
ИЗ
    Справочник.Номенклатура КАК Ном
    ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.Продажи КАК Прод
        ПО Ном.Ссылка = Прод.Номенклатура
ГДЕ
    Прод.Номенклатура ЕСТЬ NULL
```

**Ловушка:** проверять нужно поле из правой (присоединяемой) таблицы, не из левой!

### 2.2. ЛЕВОЕ СОЕДИНЕНИЕ с агрегатом во вложенном запросе

**Когда:** к основной таблице нужно присоединить агрегированные данные.

```bsl
// ПРАВИЛЬНО: продажи + сумма по каждому контрагенту
ВЫБРАТЬ
    К.Ссылка КАК Контрагент,
    ЕСТЬNULL(Прод.Сумма, 0) КАК СуммаПродаж
ИЗ
    Справочник.Контрагенты КАК К
    ЛЕВОЕ СОЕДИНЕНИЕ (
        ВЫБРАТЬ
            Прод.Контрагент КАК Контрагент,
            СУММА(Прод.Сумма) КАК Сумма
        ИЗ
            РегистрНакопления.Продажи КАК Прод
        СГРУППИРОВАТЬ ПО
            Прод.Контрагент
    ) КАК Прод
        ПО К.Ссылка = Прод.Контрагент
```

**Важно (#std655):** JOIN с подзапросом — допустим, но если данных много, лучше сначала поместить подзапрос во временную таблицу с индексом:

```bsl
// ОПТИМАЛЬНО при больших объёмах
СоздатьВременнуюТаблицу("ПродажиПоКонтрагентам", "ИНДЕКСИРОВАТЬ ПО Контрагент");
// затем JOIN с временной таблицей
```

### 2.3. Соединение с виртуальной таблицей — ЗАПРЕЩЕНО

**#std655:** Не JOIN'ить напрямую с виртуальными таблицами (Остатки, Обороты, СрезПоследних).

```bsl
// КАТЕГОРИЧЕСКИ ПЛОХО: JOIN с виртуальной таблицей
ВЫБРАТЬ
    Ном.Ссылка,
    Ост.КоличествоОстаток
ИЗ
    Справочник.Номенклатура КАК Ном
    ЛЕВОЕ СОЕДИНЕНИЕ РегистрНакопления.ТоварыНаСкладах.Остатки(
        &КонецПериода,
        Склад = &Склад
    ) КАК Ост
        ПО Ном.Ссылка = Ост.Номенклатура

// ПРАВИЛЬНО: сначала во временную таблицу с индексом
ВЫБРАТЬ
    Ост.Номенклатура КАК Номенклатура,
    Ост.КоличествоОстаток КАК КоличествоОстаток
ПОМЕСТИТЬ ВТ_Остатки
ИЗ
    РегистрНакопления.ТоварыНаСкладах.Остатки(
        &КонецПериода,
        Склад = &Склад
    ) КАК Ост
ИНДЕКСИРОВАТЬ ПО
    Номенклатура
;

ВЫБРАТЬ
    Ном.Ссылка КАК Номенклатура,
    ЕСТЬNULL(Ост.КоличествоОстаток, 0) КАК КоличествоОстаток
ИЗ
    Справочник.Номенклатура КАК Ном
    ЛЕВОЕ СОЕДИНЕНИЕ ВТ_Остатки КАК Ост
        ПО Ном.Ссылка = Ост.Номенклатура
```

### 2.4. Соединение по составному ключу

```bsl
// Соединение по двум полям
ЛЕВОЕ СОЕДИНЕНИЕ РегистрСведений.Цены КАК Цены
    ПО Т.Номенклатура = Цены.Номенклатура
    И Т.Характеристика = Цены.Характеристика
    И Т.Склад = Цены.Склад
```

**Ловушка:** порядок условий важен для оптимизатора. Сначала поля с большей селективностью.

## 3. Соединение с разыменованием составного типа

**#std654:** Обращение через точку к составному типу выполняет запрос по ВСЕМ типам регистратора.

```bsl
// ПЛОХО: Регистратор.Дата — составной тип, скрытый JOIN по всем документам
ВЫБРАТЬ
    Рег.Регистратор.Дата КАК Дата,
    Рег.Сумма КАК Сумма
ИЗ
    РегистрНакопления.Продажи КАК Рег

// ПРАВИЛЬНО: ВЫРАЗИТЬ к конкретному типу — фильтрует один документ
ВЫБРАТЬ
    ВЫРАЗИТЬ(Рег.Регистратор КАК Документ.РеализацияТоваровУслуг).Дата КАК Дата,
    Рег.Сумма КАК Сумма
ИЗ
    РегистрНакопления.Продажи КАК Рег
ГДЕ
    Рег.Регистратор ССЫЛКА Документ.РеализацияТоваровУслуг
```

**Альтернатива:** разыменование через отдельный JOIN для каждого типа:

```bsl
ВЫБРАТЬ
    Рег.Сумма КАК Сумма,
    ВЫБОР
        КОГДА Рег.Регистратор ССЫЛКА Документ.РеализацияТоваровУслуг
            ТОГДА ВЫРАЗИТЬ(Рег.Регистратор КАК Документ.РеализацияТоваровУслуг).Дата
        КОГДА Рег.Регистратор ССЫЛКА Документ.ПоступлениеТоваровУслуг
            ТОГДА ВЫРАЗИТЬ(Рег.Регистратор КАК Документ.ПоступлениеТоваровУслуг).Дата
        ИНАЧЕ NULL
    КОНЕЦ КАК Дата
ИЗ
    РегистрНакопления.Продажи КАК Рег
```

## 4. Авто-поиск пути JOIN (dependency_graph.find_join_path)

У нас в проекте реализован метод `dependency_graph.find_join_path(from, to)`, который автоматически ищет цепочку соединений между двумя объектами метаданных:

```python
from src.services.dependency_graph import MetadataDependencyGraph

graph = MetadataDependencyGraph()
path = graph.find_join_path(
    "РегистрНакопления.Продажи",
    "Справочник.Номенклатура"
)
# Вернёт: [
#   {"from": "РегистрНакопления.Продажи", "to": "РегистрСведений.КлючиАналитикиУчетаНоменклатуры",
#    "from_field": "Номенклатура", "to_field": "Номенклатура", "relation": "inner"},
#   {"from": "РегистрСведений.КлючиАналитикиУчетаНоменклатуры", "to": "Справочник.Номенклатура",
#    "from_field": "Номенклатура", "to_field": "Ссылка", "relation": "inner"}
# ]
```

Это позволяет автоматически построить JOIN-цепочку для задачи "получить данные из регистра и подтянуть связанную номенклатуру" без ручного знания графа метаданных.

## 5. Ловушки JOIN

### 5.1. Дубликаты строк при LEFT JOIN

Если в правой таблице несколько соответствий — левая строка задвоится. Решение: агрегат во вложенном подзапросе или `ОБЪЕДИНИТЬ ВСЕ` + группировка.

```bsl
// ПЛОХО: задвоит продажи, если у контрагента несколько договоров
ЛЕВОЕ СОЕДИНЕНИЕ Справочник.ДоговорыКонтрагентов КАК Д
    ПО К.Ссылка = Д.Владелец

// ПРАВИЛЬНО: один договор по условию (например, основной)
ЛЕВОЕ СОЕДИНЕНИЕ (
    ВЫБРАТЬ ПЕРВЫЕ 1
        Д.Владелец КАК Владелец,
        Д.Ссылка КАК Ссылка
    ИЗ Справочник.ДоговорыКонтрагентов КАК Д
    ГДЕ Д.Владелец = &Контрагент И Д.ПометкаУдаления = ЛОЖЬ
    УПОРЯДОЧИТЬ ПО Д.Основной УБЫВ
) КАК Д
    ПО К.Ссылка = Д.Владелец
```

### 5.2. ИЛИ в условии соединения (#std655)

```bsl
// ПЛОХО: ИЛИ в условии JOIN — нет индекса
ЛЕВОЕ СОЕДИНЕНИЕ Таблица2 КАК Т2
    ПО Т1.А = Т2.А ИЛИ Т1.Б = Т2.Б

// ПРАВИЛЬНО: два LEFT JOIN + ОБЪЕДИНИТЬ
ВЫБРАТЬ ... ИЗ Т1 ЛЕВОЕ СОЕДИНЕНИЕ Т2 ПО Т1.А = Т2.А
ОБЪЕДИНИТЬ ВСЕ
ВЫБРАТЬ ... ИЗ Т1 ЛЕВОЕ СОЕДИНЕНИЕ Т2 ПО Т1.Б = Т2.Б
```

### 5.3. NULL в условии соединения

```bsl
// ПЛОХО: NULL != NULL, строки потеряются
ЛЕВОЕ СОЕДИНЕНИЕ Т2 ПО Т1.Поле = Т2.Поле

// ПРАВИЛЬНО: явная обработка NULL
ЛЕВОЕ СОЕДИНЕНИЕ Т2 ПО
    (Т1.Поле = Т2.Поле
     ИЛИ (Т1.Поле ЕСТЬ NULL И Т2.Поле ЕСТЬ NULL))
```

### 5.4. Функции в условии JOIN

```bsl
// ПЛОХО: функция блокирует индекс
ЛЕВОЕ СОЕДИНЕНИЕ Т2 ПО НАЧАЛОПЕРИОДА(Т1.Дата, ДЕНЬ) = НАЧАЛОПЕРИОДА(Т2.Дата, ДЕНЬ)

// ПРАВИЛЬНО: диапазон вместо функции
ЛЕВОЕ СОЕДИНЕНИЕ Т2 ПО
    Т2.Дата МЕЖДУ &НачалоДня И &КонецДня
```

## 6. Стратегия: временные таблицы vs. JOIN

| Ситуация | Что лучше |
|----------|-----------|
| Мало строк (< 1000) в правой таблице | Прямой JOIN |
| Много строк (> 10000), многократное использование | Временная таблица с индексом |
| Виртуальная таблица (Остатки/Обороты) | **Только** временная таблица |
| Подзапрос используется в нескольких JOIN | Временная таблица |
| Однократный JOIN с простой агрегацией | Подзапрос (временная таблица лишняя) |

### Паттерн "временная таблица + индекс"

```bsl
ВЫБРАТЬ
    Прод.Номенклатура КАК Номенклатура,
    Прод.Склад КАК Склад,
    СУММА(Прод.Сумма) КАК Сумма
ПОМЕСТИТЬ ВТ_Продажи
ИЗ
    РегистрНакопления.Продажи КАК Прод
ГДЕ
    Прод.Период МЕЖДУ &Начало И &Конец
СГРУППИРОВАТЬ ПО
    Прод.Номенклатура, Прод.Склад
ИНДЕКСИРОВАТЬ ПО
    Номенклатура, Склад
;

// Дальше можно многократно JOIN'ить с ВТ_Продажи
```

## 7. План выполнения запроса — на что смотреть

После `Запрос.Выполнить().План()`:

| Признак | Проблема |
|---------|----------|
| `SCAN` по большой таблице | Нет индекса по условию WHERE |
| `SEARCH` с `OR` в условии | ИЛИ блокирует индекс |
| `MULTI-INDEX SCAN` по составному типу | Не сделано ВЫРАЗИТЬ() |
| `JOIN ... SUBQUERY` | Подзапрос не во временной таблице |
| `TEMP TABLE` без `INDEXED BY` | Временная таблица без индекса |
| Сортировка на сервере без `УПОРЯДОЧИТЬ ПО` | Сортировка в коде вместо запроса |

## 9. Шаблоны для `generate_query`

### 9.1. Базовый LEFT JOIN + агрегат

```bsl
ВЫБРАТЬ
    А.Поле КАК Поле,
    ЕСТЬNULL(Б.Агрегат, 0) КАК Агрегат
ИЗ
    ТаблицаА КАК А
    ЛЕВОЕ СОЕДИНЕНИЕ (
        ВЫБРАТЬ
            Б.Ключ КАК Ключ,
            СУММА(Б.Поле) КАК Агрегат
        ИЗ ТаблицаБ КАК Б
        ГДЕ Б.Период МЕЖДУ &Начало И &Конец
        СГРУППИРОВАТЬ ПО Б.Ключ
    ) КАК Б
        ПО А.Ключ = Б.Ключ
```

### 9.2. Соединение с остатками через временную таблицу

```bsl
ВЫБРАТЬ
    Ост.Номенклатура КАК Номенклатура,
    Ост.Склад КАК Склад,
    Ост.КоличествоОстаток КАК Количество
ПОМЕСТИТЬ ВТ_Остатки
ИЗ
    РегистрНакопления.ТоварыНаСкладах.Остатки(
        &КонецПериода,
        Склад = &Склад
    ) КАК Ост
ИНДЕКСИРОВАТЬ ПО
    Номенклатура, Склад
;

ВЫБРАТЬ
    Ном.Ссылка КАК Номенклатура,
    ЕСТЬNULL(Ост.Количество, 0) КАК Количество
ИЗ
    Справочник.Номенклатура КАК Ном
    ЛЕВОЕ СОЕДИНЕНИЕ ВТ_Остатки КАК Ост
        ПО Ном.Ссылка = Ост.Номенклатура
```

### 9.3. Анти-JOIN (строки без соответствия)

```bsl
ВЫБРАТЬ
    А.Поле КАК Поле
ИЗ
    ТаблицаА КАК А
    ЛЕВОЕ СОЕДИНЕНИЕ ТаблицаБ КАК Б
        ПО А.Ключ = Б.Ключ
ГДЕ
    Б.Ключ ЕСТЬ NULL
```
