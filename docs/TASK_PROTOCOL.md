# Протокол решения задач 1С

> **Обязательный документ.** При любой задаче по 1С-разработке (не по разработке самого проекта `1c-ai-dev-env`) ассистент соблюдает этот протокол.

## Суть

Когда пользователь ставит задачу по 1С (создать справочник, написать обработку, доработать модуль, исправить баг в .bsl коде) — ассистент не пишет код сразу. Он прогоняет задачу через **5 ролей**, каждая из которых имеет чёткие артефакты и чек-листы.

Это работает в режиме прямого общения с ассистентом (текущий чат, IDE с system prompt, CLI). Для MCP-режима (Cursor/Claude) — LLM оркеструет сама, протокол служит ориентиром.

---

## 5 ролей

### 🧠 Архитектор (Architect)

**Цель:** Понять задачу, собрать контекст, спроектировать решение.

**Вход:** Описание задачи от пользователя.

**Выход:** План решения — что создаём, какие метаданные, какие модули, какие методы платформы используем, какие стандарты 1С соблюдаем.

**Что делает:**
1. Вызывает `solve_context(query, config)` — собирает контекст: методы платформы (BM25 поиск), API-справочник конфигурации, стандарты 1С
2. Если нужно — уточняет у пользователя детали (версия конфигурации, требования, ограничения)
3. Определяет тип задачи: новая фича / bugfix / рефакторинг / доработка типовой
4. Проектирует структуру:
   - Метаданные (какие объекты, их реквизиты, табличные части)
   - Модули (общие модули, модули объектов, формы)
   - Интеграции (запросы, внешние источники, HTTP-сервисы)
5. Выбирает методы платформы из контекста (не изобретает велосипед)
6. Определяет стандарты 1С, которые критичны для этой задачи

**Чек-лист Архитектора:**
- [ ] `solve_context` вызван, контекст получен
- [ ] Тип задачи определён (фича / bugfix / рефакторинг)
- [ ] Метаданные спроектированы (если применимо)
- [ ] Модули определены (какие .bsl файлы нужны)
- [ ] Методы платформы выбраны из контекста (не выдуманы)
- [ ] Стандарты 1С к соблюдению перечислены
- [ ] План записан и согласован с пользователем

**Запреты:**
- ❌ Не пишет код (это задача Программиста)
- ❌ Не пропускает `solve_context` — даже для «простой» задачи
- ❌ Не придумывает методы платформы — только из результатов поиска

---

### 👨‍💻 Программист (Programmer)

**Цель:** Реализовать код по плану Архитектора.

**Вход:** План от Архитектора (метаданные, модули, методы, стандарты).

**Выход:** `.bsl` файлы + XML метаданных.

**Что делает:**
1. По плану Архитектора создаёт структуру модулей:
   - `#Область ПрограммныйИнтерфейс` — экспортные процедуры/функции
   - `#Область СлужебныйПрограммныйИнтерфейс` — служебные экспортные
   - `#Область СлужебныеПроцедурыИФункции` — внутренние
   - `#Область Инициализация` — обработчики
2. Использует методы платформы из плана (с правильным синтаксисом из `solve_context`)
3. Соблюдает стандарты 1С из плана:
   - БСП-подходы (`ОбщегоНазначения.`, `ОбщегоНазначенияКлиентСервер.`)
   - Запросы через `Запрос.УстановитьПараметр`, не конкатенация
   - Проверка прав через `ПравоДоступа`
   - Транзакции только на сервере
4. Пишет XML метаданных (если создаются новые объекты)
5. Документирует экспортные процедуры (см. Документатор)

**Чек-лист Программиста:**
- [ ] Структура модуля: 4 области (#Область ПрограммныйИнтерфейс / Служебный...)
- [ ] Методы платформы из плана использованы (не выдуманы)
- [ ] Запросы — через `УстановитьПараметр`, без конкатенации
- [ ] Серверные вызовы минимизированы
- [ ] Транзакции — только в `&НаСервере` (если применимо)
- [ ] Имена переменных — осмысленные, не `А`, `Б`, `Перем1`
- [ ] XML метаданных создан (если новые объекты)

**Запреты:**
- ❌ Не отклоняется от плана Архитектора без согласования
- ❌ Не использует методы, которых не было в контексте Архитектора
- ❌ Не пишет `Сообщить()`, `Выполнить()`, `Вычислить()` (запрещено стандартами)
- ❌ Не использует точечную нотацию (Товар.Цена) — только через `Ссылка.Свойство`

### 🎨 Стилист (Stylist)

**Цель:** Проверить код на соответствие стандартам 1С и читаемость.

**Вход:** Код от Программиста.

**Выход:** Список правок + исправленный код.

**Что делает:**
1. Запускает `check_standards(file_path)` — 56 правил (стилистика, антипаттерны, запросы, клиент-сервер, безопасность)
2. Запускает `analyze_bsl(file_path)` — 187 диагностик BSL LS (если Java установлена)
3. Анализирует читаемость:
   - Длина строк (не больше 120 символов)
   - Вложенность (не больше 3 уровней)
   - Имена (осмысленные, без сокращений вроде `О`, `Пер`)
   - Комментарии на русском, код на русском (в 1С)
4. Проверяет структуру модуля (правильное расположение областей)
5. Проверяет em-dash (—), ё, венгерскую нотацию, закомментированный код

**Чек-лист Стилиста:**
- [ ] `check_standards` вызван, 0 errors
- [ ] `analyze_bsl` вызван (или отмечено, что BSL LS не установлен)
- [ ] Структура областей корректна (ПрограммныйИнтерфейс → Служебный... → ... → Инициализация)
- [ ] Длина строк ≤ 120 символов
- [ ] Вложенность ≤ 3 уровней
- [ ] em-dash (—) вместо дефиса в комментариях
- [ ] Нет закомментированного кода
- [ ] Нет `Сообщить()`, `Выполнить()`, `Вычислить()`
- [ ] Нет точечной нотации в запросах
- [ ] Код читается без «зависаний» — логика прозрачна

**Запреты:**
- ❌ Не меняет логику кода (только стиль)
- ❌ Не добавляет новый функционал
- ❌ Не удаляет тесты
- ❌ Если `check_standards` вернул errors — обязательно исправляет, не игнорирует

---

### 📝 Документатор (Documenter)

**Цель:** Снабдить код документацией, обновить документацию проекта.

**Вход:** Стилизованный код от Стилиста.

**Выход:** Документированный код + примеры использования.

**Что делает:**
1. Для каждой экспортной процедуры/функции пишет docstring:
   ```
   // Создаёт новый элемент справочника Товары.
   //
   // Параметры:
   //   Наименование - Строка - наименование товара.
   //   Артикул - Строка - артикул производителя.
   //
   // Возвращаемое значение:
   //   СправочникСсылка.Товары - ссылка на созданный элемент.
   //
   // Пример:
   //   Ссылка = СоздатьТовар("Молоток", "M-001");
   ```
2. Для сложной логики — поясняющие комментарии
3. Обновляет `docs/` если задача меняет архитектуру
4. Обновляет описание конфигурации (если применимо)
5. Создаёт пример использования (snippet)

**Чек-лист Документатора:**
- [ ] Все экспортные процедуры/функции имеют docstring
- [ ] В docstring: Назначение, Параметры, Возвращаемое значение, Пример
- [ ] Сложная логика пояснена комментариями
- [ ] Пример использования работает (мысленно прогнан)
- [ ] Документация конфигурации обновлена (если нужно)

**Запреты:**
- ❌ Не пишет docstring для служебных процедур (только для экспортных)
- ❌ Не дублирует то, что очевидно из кода
- ❌ Не пишет docstring после того, как код уже отдан Проверяющему

---

### ✅ Проверяющий (Verifier)

**Цель:** Финальная проверка — код готов к коммиту.

**Вход:** Документированный код.

**Выход:** Отчёт: total_errors, total_warnings, verdict.

**Что делает:**
1. Запускает `solve_check(file_path, level='full')` — полная проверка:
   - `check_1c_standards` (56 правил)
   - `analyze_bsl` (187 диагностик BSL LS)
   - `check_metadata_standards` (18 правил XML, если есть метаданные)
2. Проверяет acceptance criteria из плана Архитектора:
   - Все ли метаданные созданы?
   - Все ли модули написаны?
   - Все ли методы платформы из плана использованы?
3. Прогоняет regression (если менялся существующий код):
   - Не сломал ли другие модули?
   - Не изменил ли public API?
4. Формирует итоговый отчёт:
   - `verdict: ready` — 0 errors, 0 warnings → код готов
   - `verdict: warnings` — 0 errors, есть warnings → код годен, но лучше поправить
   - `verdict: errors` — есть errors → код НЕ готов, вернуть Программисту

**Чек-лист Проверяющего:**
- [ ] `solve_check --level full` вызван
- [ ] verdict = `ready` или `warnings` (не `errors`)
- [ ] Acceptance criteria из плана Архитектора выполнены
- [ ] Regression: существующий код не сломан
- [ ] Отчёт передан пользователю с рекомендацией

**Запреты:**
- ❌ Не пропускает код с `verdict: errors`
- ❌ Не пишет новый код (возвращает Программисту)
- ❌ Не пропускает regression-нарушения

## 3 протокола запуска

### Протокол A: Полный (сложные задачи)

**Триггеры:**
- Новая фича (справочник, документ, обработка с нуля)
- Рефакторинг модуля
- Breaking change (меняет public API)
- Интеграция (HTTP-сервис, внешний источник)
- Доработка типовой конфигурации (через расширение)

**Роли:** 🧠 Архитектор → 👨‍💻 Программист → 🎨 Стилист → 📝 Документатор → ✅ Проверяющий

### Протокол B: Стандартный (средние задачи)

**Триггеры:**
- Bugfix в существующем коде
- Небольшая доработка (добавить реквизит, процедуру)
- Оптимизация производительности
- Исправление диагностики BSL LS

**Роли:** 👨‍💻 Программист → 🎨 Стилист → ✅ Проверяющий

### Протокол C: Быстрый (тривиальные задачи)

**Триггеры:**
- Опечатка в комментарии
- Обновление версии в `Configuration.xml`
- Переименование переменной
- Добавление пропущенного `Экспорт`

**Роли:** 👨‍💻 Программист → ✅ Проверяющий (быстрая: только `check_standards`)

## Алгоритм выбора протокола

```
Получил задачу по 1С
    │
    ├─ Это задача по проекту 1c-ai-dev-env? → НЕ применяем протокол (это разработка самого инструмента)
    │
    ├─ Это задача по 1С (создать объект, написать .bsl, доработать конфигурацию)?
    │   │
    │   ├─ Новая фича / рефакторинг / breaking change? → Протокол A (полный)
    │   │
    │   ├─ Bugfix / небольшая доработка? → Протокол B (стандартный)
    │   │
    │   └─ Опечатка / переименование / версия? → Протокол C (быстрый)
    │
    └─ Это вопрос / обсуждение / анализ? → Протокол не нужен (просто отвечаю)
```

## Формат работы

### Объявление протокола

При получении задачи ассистент **обязан** первым сообщением объявить:

```
📋 Задача: <краткая формулировка>
🎯 Протокол: A (Полный) — т.к. <причина выбора>

Роли:
  🧠 Архитектор     — собираю контекст, проектирую
  👨‍💻 Программист   — пишу код
  🎨 Стилист        — проверяю стандарты
  📝 Документатор   — документирую
  ✅ Проверяющий     — финальная проверка
```

### Объявление роли

Перед выполнением каждой роли:

```
🧠 [Архитектор] Анализирую задачу...
   → Вызываю solve_context("создать справочник Товары", "ut11")
   → Анализирую методы платформы...
   → Проектирую структуру...

📐 План:
  Метаданные:
    - Справочник.Товары (иерархия групп и элементов)
    - Реквизиты: Артикул, ЕдиницаИзмерения, ЦенаПоУмолчанию
    - Табличная часть: Цены (Период, Цена)
  
  Модули:
    - Модуль менеджера: СоздатьТовар(), НайтиПоАртикулу()
    - Модуль объекта: ПередЗаписью() — проверка артикула
  
  Методы платформы:
    - Справочники.Товары.СоздатьЭлемент()
    - СправочникМенеджер.НайтиПоКоду()
  
  Стандарты к соблюдению:
    - STD 454: Структура модуля (4 области)
    - STD 455: Экспортные в ПрограммныйИнтерфейс
    - STD 12: Транзакции на сервере
```

### Переход к следующей роли

```
✅ [Архитектор] План готов. Передаю Программисту.

👨‍💻 [Программист] Реализую по плану...
   → Создаю МодульМенеджера.bsl
   → Создаю МодульОбъекта.bsl
   ...
```

### Финальный отчёт Проверяющего

```
✅ [Проверяющий] Финальная проверка
   → solve_check --level full
   → Результат: 0 errors, 2 warnings, verdict=warnings

📊 Итог:
  - Метаданные: Справочник.Товары ✅
  - Модули: 2 файла созданы ✅
  - Методы платформы: 2 использованы ✅
  - Стандарты: 56 проверены, 0 errors ✅
  - BSL LS: 187 диагностик, 0 errors, 2 warnings ⚠️
  - Regression: не применимо (новый код)

⚠️ Warnings (не критично):
  - BSL_LS: код 100 → ИспользоватьЗначениеПоУмолчанию (рекомендация)

🎯 Verdict: warnings — код готов к коммиту, warnings можно исправить позже.
```

## Anti-patterns (что НЕ делать)

1. **Писать код сразу.** Сначала Архитектор собирает контекст.
2. **Пропускать роли.** Даже для «быстрой» задачи — Программист → Проверяющий.
3. **Использовать методы платформы по памяти.** Только из `solve_context`.
4. **Игнорировать `verdict: errors`.** Возвращать Программисту.
5. **Менять логику в роли Стилиста.** Только стиль.
6. **Писать docstring для служебных процедур.** Только для экспортных.
7. **Не объявлять роль.** Пользователь должен видеть, какая роль работает.
