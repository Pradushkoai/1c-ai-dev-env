tools/ — определения MCP tools по категориям.

P1.3: вынесено из src/mcp_server.py для декомпозиции (SRP).

Каждый модуль экспортирует: - DESCRIPTIONS: list[dict] — статическое описание tools для _get_tools_description()

Категории: - search: поиск по методам платформы и коду конфигурации - analyzers: анализ BSL кода (11 анализаторов) - metadata: метаданные конфигурации (объекты, формы, СКД) - generate: генерация кода (обработки, отчёты, EPF) - dsl_cfe: DSL компиляторы и работа с CFE - quality: проверка качества (формы, СКД, diff) - context: сбор контекста (solve, knowledge, data_status) - misc: прочие (inspect, openspec, depgraph, skd_trace)

_security.py — общие security-утилиты для MCP handlers.

Главная задача: защита от path traversal (CWE-22) при работе с file_path, которые приходят от MCP-клиента (IDE/LLM). Без валидации атакующий может передать file_path=’../../../../etc/passwd’ и получить содержимое любого файла, доступного процессу MCP-сервера.

См. audit P1.8: src/mcpserver/handlers/quality.py:83-91 — path traversal.

src.mcpserver.handlers._security.resolve_path_within_project(raw_path, project, *, must_exist=False)[source]¶

Резолвит путь, проверяя что он находится внутри project.paths.root.

Алгоритм: 1. Если путь относительный — он считается относительным к project.paths.root

(это поведение сохранено из оригинальных handlers для обратной совместимости).

  1. Резолвим через os.path.realpath() чтобы раскрыть ‘..’ и симлинки.

  2. Проверяем, что полученный абсолютный путь начинается с project.paths.root.

  3. Если must_exist=True — дополнительно проверяем существование.

Parameters:

  • raw_path (str) – Путь от MCP-клиента (может быть относительным, абсолютным, содержать ‘..’ или симлинки).

  • project (Project) – Project для получения корня.

  • must_exist (bool) – Если True, вернуть None если путь не существует.

Return type:

Path | None

Returns:

Path если путь валиден и находится внутри проекта, иначе None.

Examples

>>> resolve_path_within_project("data/file.bsl", project)
PosixPath('/repo/data/file.bsl')

>>> resolve_path_within_project("../../etc/passwd", project)
None  # path traversal blocked

>>> resolve_path_within_project("/etc/passwd", project)
None  # absolute path outside project

async_helpers.py — утилиты для async-safe вызовов sync-кода в MCP handlers.

P1.10: MCP-сервер работает в event loop asyncio. Sync-вызовы процессора (TaskProcessor.solve, TaskProcessor.check, project.search_methods) блокируют event loop — параллельные MCP-запросы от IDE стоят в очереди.

Решение: оборачивать sync-вызовы в asyncio.to_thread(), который запускает их в отдельном потоке. event loop остаётся свободным для других запросов.

async src.mcpserver.handlers._async_helpers.run_sync(func, /, *args, **kwargs)[source]¶

Запустить sync-функцию в отдельном потоке, не блокируя event loop.

Обёртка над asyncio.to_thread() с более коротким синтаксисом и поддержкой kwargs (через functools.partial под капотом).

Parameters:

  • func (Callable[..., TypeVar(T)]) – Синхронная функция для выполнения.

  • *args (Any) – Аргументы для func.

  • **kwargs –

    Аргументы для func.

Return type:

TypeVar(T)

Returns:

Результат func(*args, **kwargs).

Examples

# Вместо: # result = processor.check(Path(file_path), level=”standard”) # Делаем: # result = await run_sync(processor.check, Path(file_path), level=”standard”)

# С lambda для сложных выражений: # configs = await run_sync(lambda: project.list_configs_info())

src.mcpserver.handlers._async_helpers.sync_to_async(func)[source]¶

Декоратор: преобразует sync-функцию в async-функцию через asyncio.to_thread.

Удобно для создания async-обёрток над sync-методами:

# В handler: async_search = sync_to_async(project.search_methods) results = await async_search(query, limit=10)
