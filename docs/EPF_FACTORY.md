# EPF Factory — создание внешних обработок 1С без платформы

> **Инструмент полного цикла**: BSL-код → готовый `.epf` файл с проверкой через BSL LS и round-trip валидацией. **Не требует установленной 1С.**

## Быстрый старт

### Через CLI

```bash
# Минимальный пример — пустая обработка
1c-ai epf-factory create \
    --name "МояОбработка" \
    --bsl module.bsl \
    --output /tmp/МояОбработка.epf
```

`module.bsl` — это BSL-код **модуля формы** (не модуля объекта). Минимальный шаблон:

```bsl
#Область ПрограммныйИнтерфейс

#КонецОбласти

#Область СлужебныеПроцедурыИФункции

&НаСервере
Процедура ПриСозданииНаСервере(Отказ, СтандартнаяОбработка)
    // Ваш код
КонецПроцедуры

#КонецОбласти
```

### Через MCP (из Cursor, Claude Desktop и т.д.)

Вызовите инструмент `epf_factory_create`:

```json
{
  "name": "epf_factory_create",
  "arguments": {
    "name": "МояОбработка",
    "synonym": "Моя обработка",
    "bsl_code": "#Область ПрограммныйИнтерфейс\n\n#КонецОбласти\n",
    "output_path": "/tmp/МояОбработка.epf"
  }
}
```

## Архитектура

```
┌────────────────────────────────────────────────────────────────┐
│                    EpfFactory.create_epf()                     │
├────────────────────────────────────────────────────────────────┤
│  1. Копирование шаблонов v8unpack (templates/epf_factory/)     │
│  2. Генерация UUID (proc_uuid, form_uuid, file_uuid)           │
│  3. Патч ExternalDataProcessor.json (11 мест: header, copyinfo)│
│  4. Патч Form.id.json (новый UUID формы)                       │
│  5. Запись Form.obj.bsl (BSL-код модуля формы)                 │
│  6. Проверка BSL через BSL Language Server (опционально)       │
│  7. Сборка .epf через v8unpack -B                              │
│  8. Проверка round-trip: распаковка и сравнение BSL            │
└────────────────────────────────────────────────────────────────┘
                                │
                                ▼
                    ┌───────────────────────┐
                    │  Готовый .epf файл    │
                    │  (валидная сигнатура  │
                    │   1С-контейнера)      │
                    └───────────────────────┘
```

### Компоненты

| Компонент | Источник | Назначение |
|---|---|---|
| `templates/epf_factory/` | Извлечено из реального EPF через `v8unpack -E` | 4 шаблона v8unpack-формата (скелеты JSON) |
| `v8unpack` (saby, Python) | `requirements.txt` | Pack/unpack бинарного `.epf`-контейнера |
| BSL Language Server | `/home/z/.local/bin/bsl-language-server` | Статический анализ BSL без 1С |
| `src/services/epf_factory.py` | Новый, 470 строк | Оркестрация всех шагов |

### Шаблоны (4 файла)

| Файл | Размер | Назначение |
|---|---|---|
| `ExternalDataProcessor.template.json` | 3.6 КБ | Метаданные обработки: name, synonym, UUID, header, copyinfo |
| `Form.template.json` | 10 КБ | Метаданные формы: name, FormType=Managed, UsePurposes (Platform+Mobile) |
| `Form.id.template.json` | 52 байта | UUID формы (placeholder) |
| `Form.elem.empty.json` | 1 КБ | Пустая форма: только реквизит «Объект», без элементов |

## CLI

### Создание EPF

```bash
1c-ai epf-factory create \
    --name "ИмяОбработки" \
    --synonym "Синоним для отображения" \
    --bsl path/to/module.bsl \
    --output /path/to/output.epf \
    [--form-name "Форма"] \
    [--save-sources] \
    [--skip-bsl-validation]
```

**Пример:**

```bash
1c-ai epf-factory create \
    --name "ВыгрузкаНоменклатурыВExcel" \
    --synonym "Выгрузка номенклатуры в Excel" \
    --bsl ./src/ВыгрузкаНоменклатуры.bsl \
    --output ./build/ВыгрузкаНоменклатуры.epf
```

Вывод:

```

   Синоним: Выгрузка номенклатуры в Excel
   UUID обработки: 852185f5-b31f-4508-9cf4-e2c90e61a11a
```

## Python API

```python
from src.services.epf_factory import EpfFactory

factory = EpfFactory()
result = factory.create_epf(
    name="МояОбработка",
    synonym="Моя обработка",
    bsl_code=open("module.bsl", encoding="utf-8").read(),
    output_epf="/tmp/МояОбработка.epf",
    form_name="Форма",           # опционально
    save_sources=False,          # опционально — сохранить v8unpack-исходники
    skip_bsl_validation=False,   # опционально — пропустить BSL LS
)

if result.ok:
    print(f"EPF создан: {result.epf_path}")
    print(f"Размер: {result.size_bytes} байт")
    print(f"BSL: {result.bsl_lines} строк, {result.bsl_errors} errors")
    print(f"Round-trip: {'OK' if result.round_trip_ok else 'FAIL'}")
else:
    print(f"Ошибка: {result.error}")
```

### Атрибуты результата (`EpfFactoryResult`)

| Атрибут | Тип | Описание |
|---|---|---|
| `ok` | bool | Успех операции |
| `error` | str | Текст ошибки (если `ok=False`) |
| `epf_path` | Path | Путь к созданному .epf |
| `size_bytes` | int | Размер файла |
| `name` | str | Имя обработки |
| `synonym` | str | Синоним |
| `proc_uuid` | str | UUID обработки |
| `form_uuid` | str | UUID формы |
| `bsl_lines` | int | Кол-во строк в BSL-модуле |
| `bsl_warnings` | int | Кол-во warnings от BSL LS |
| `bsl_errors` | int | Кол-во errors от BSL LS |
| `round_trip_ok` | bool | Round-trip проверка (распаковка + сравнение BSL) |
| `work_dir` | Path | Каталог с исходниками (если `save_sources=True`) |

## MCP-инструменты

### `epf_factory_create`

Создать `.epf` из BSL-кода.

**Параметры:**

| Параметр | Тип | Обязательный | Описание |
|---|---|---|---|
| `name` | string | ✅ | Имя обработки (латиница/кириллица, без пробелов) |
| `output_path` | string | ✅ | Путь к выходному .epf файлу |
| `synonym` | string | — | Синоним (по умолчанию = name) |
| `bsl_code` | string | — | BSL-код модуля формы (inline) |
| `bsl_path` | string | — | Путь к .bsl файлу (альтернатива `bsl_code`) |
| `form_name` | string | — | Имя формы (по умолчанию "Форма") |
| `skip_bsl_validation` | boolean | — | Пропустить проверку BSL LS |
| `save_sources` | boolean | — | Сохранить v8unpack-исходники |

Если ни `bsl_code`, ни `bsl_path` не заданы — используется минимальный шаблон.

**Пример вызова из Claude Desktop:**

```
Используй инструмент epf_factory_create чтобы создать обработку:
- name: "СписокЗаказов"
- synonym: "Список заказов клиентов"  
- bsl_code: (твой BSL-код)
- output_path: "/tmp/СписокЗаказов.epf"
```

### `epf_factory_templates`

Возвращает список доступных шаблонов (без параметров).

## Параметры

### Имя обработки (`name`)

- Латиница, кириллица, цифры, подчеркивание
- **Без пробелов**
- Должно быть валидным идентификатором 1С

✅ `МояОбработка`, `ВыгрузкаNomенклатуры`, `Sync_HTTP_Orders`
❌ `Моя Обработка`, `Моя-Обработка`, `1Обработка`

### Синоним (`synonym`)

- Произвольная строка
- Отображается в интерфейсе 1С
- По умолчанию = `name`

### BSL-код

- Это **модуль формы**, не модуль объекта
- Рекомендуется использовать стандартные области:
  - `#Область ПрограммныйИнтерфейс`
  - `#Область СлужебныйПрограммныйИнтерфейс`
  - `#Область СлужебныеПроцедурыИФункции`
  - `#Область ОбработчикиСобытийФормы`
- Используйте **табы** для отступов (STD 456)
- Без буквы `ё` в коде (STD 456)
- Без EM DASH (`—`), используйте дефис (`-`)

### Описание формы (`form_spec`)

Универсальный DSL для описания структуры формы — какие нужны реквизиты,
какие колонки в таблицах значений. Это **обязательно** если в BSL вы
обращаетесь к реквизитам формы по имени (например, `ТаблицаСписка.Добавить()`)
— иначе компилятор 1С выдаст «Переменная не определена».

**Структура DSL:**

```json
{
  "props": [
    {
      "name": "ИмяРеквизита",
      "type": "Тип",
      "synonym": "Синоним для отображения",
      "...": "..."
    }
  ]
}
```

**Поддерживаемые типы реквизитов:**

| Тип | Описание | Доп. параметры |
|---|---|---|
| `DataProcessorObject` | Объект внешней обработки (обязательный, обычно id=1) | — |
| `ValueTable` | Таблица значений с колонками | `columns`: список колонок |
| `Date` | Дата | — |
| `String` | Строка | `length`: длина (по умолчанию 50) |
| `Number` | Число | `digits`: всего знаков, `fraction`: дробных |
| `Boolean` | Булево | — |
| `CatalogRef` | СправочникСсылка | `catalog`: имя справочника |
| `DocumentRef` | ДокументСсылка | `document`: имя документа |

**Пример `form_spec` (форма с таблицей списка документов):**

```json
{
  "props": [
    {"name": "Объект", "type": "DataProcessorObject"},
    {
      "name": "ТаблицаСписка",
      "type": "ValueTable",
      "synonym": "Список обходов",
      "columns": [
        {"name": "Дата", "type": "Date", "synonym": "Дата"},
        {"name": "Номер", "type": "String", "length": 50, "synonym": "Номер"},
        {"name": "ОтветственныйЗаОбход", "type": "String", "length": 150,
         "synonym": "Ответственный за обход"},
        {"name": "ВремяСтарта", "type": "Date", "synonym": "Время старта"},
        {"name": "ВремяОкончания", "type": "Date", "synonym": "Время окончания"},
        {"name": "УведомленияСформированы", "type": "Boolean",
         "synonym": "Уведомления сформированы"},
        {"name": "ВсеМеткиОтсканированы", "type": "Boolean",
         "synonym": "Все метки отсканированы"},
        {"name": "ОтправленоНаСервер", "type": "Boolean",
         "synonym": "Отправлено на сервер"}
      ]
    }
  ]
}
```

**Пример `form_spec` (простая форма с несколькими полями):**

```json
{
  "props": [
    {"name": "Объект", "type": "DataProcessorObject"},
    {"name": "ДатаНачала", "type": "Date", "synonym": "Дата начала"},
    {"name": "ДатаОкончания", "type": "Date", "synonym": "Дата окончания"},
    {"name": "Комментарий", "type": "String", "length": 200, "synonym": "Комментарий"},
    {"name": "Сумма", "type": "Number", "digits": 15, "fraction": 2, "synonym": "Сумма"}
  ]
}
```

**Передача `form_spec`:**

- CLI: `--form-spec path/to/form_spec.json`
- Python API: `factory.create_epf(..., form_spec={"props": [...]})`
- MCP: параметр `form_spec` (inline object) или `form_spec_path` (путь к файлу)

## Что внутри EPF

Структура v8unpack-исходников после сборки:

```
src/
├── ExternalDataProcessor.json          # Метаданные обработки
└── Form/
    └── Форма/
        ├── Form.json                   # Метаданные формы
        ├── Form.id.json                # UUID формы
        ├── Form.elem.json              # Элементы формы (пустая)
        └── Form.obj.bsl                # BSL-модуль формы
```

После сборки в `.epf`:

```
.epf (8-10 КБ)
├── Сигнатура: ff ff ff 7f (1С-контейнер)
├── root
├── version
├── versions
├── copyinfo
├── <proc_uuid>          # Метаданные обработки
├── <form_uuid>          # Метаданные формы
└── <form_uuid>.0        # Данные формы (BSL + элементы)
```

## Ограничения

1. **Внешний вид формы**: элементы формы (кнопки, поля, таблицы) нужно создавать **программно** в BSL через `Элементы.Добавить(...)` в `ПриСозданииНаСервере`. Конвертер из EDT Form.xml → v8unpack Form.elem.json пока не реализован.

2. **Мобильное приложение**: тип `ПараметрыФормыДинамическогоСписка` недоступен. Используйте `ТаблицаЗначений` + запрос (см. пример `spisok_bsl_module.py`).

3. **Множественные формы**: сейчас создаётся одна форма (по умолчанию `Форма`). Для нескольких форм нужно расширять шаблон.

4. **Макеты, СКД, роли**: не входят в базовый шаблон. Можно добавить через `1c-epf-add-form` из `claude-code-skills-1c` или руками.

5. **BSL LS требует Java**: если Java не установлена, используйте `--skip-bsl-validation`.

6. **Реквизиты формы — статически**: в BSL нельзя обращаться к реквизитам, созданным через `ИзменитьРеквизиты()` в runtime — компилятор 1С их не видит. Все реквизиты формы, к которым обращаетсяесь в BSL, **обязательно** объявляйте в `form_spec`.

## ⚠️ Что НЕ проверяет тестирование

`epf-factory` проверяет **структуру** EPF, но НЕ проверяет **семантику**:

| Что проверяет | Что НЕ проверяет |
|---|---|
| ✅ Сигнатура 1С-контейнера (`ffffff7f`) | ❌ Откроется ли в 1С без ошибок |
| ✅ Round-trip (распаковка/сборка) | ❌ Видит ли компилятор 1С все реквизиты |
| ✅ BSL LS — синтаксис и стиль (187 диагностик) | ❌ Совместимость с мобильным приложением |
| ✅ Стандарты 1С (56 правил, СТД 456) | ❌ Существуют ли указанные типы в конфигурации |
| ✅ Структура v8unpack-исходников | ❌ Правильно ли описаны `Pattern` для типов |

**Финальная проверка только в 1С**: открыть `.epf` через `Файл → Открыть` и убедиться, что:
- Нет ошибок компиляции
- Форма открывается
- Реквизиты видны
- Запросы выполняются
- Элементы формы создаются

**Рекомендация**: после сборки `.epf` через `epf-factory` всегда открывайте его в 1С перед отправкой заказчику.

## Решение проблем

### `Round-trip: ❌ FAIL`

Распаковка собранного `.epf` дала BSL-модуль, отличный от исходного. Возможные причины:
- BSL-код содержит символы, которые v8unpack нормализует (например, разные виды переводов строк)
- BSL-код слишком большой (> 64 КБ после сериализации)

**Решение**: EPF всё равно валиден и открывается в 1С, просто round-trip проверка не прошла. Можно проигнорировать или уменьшить BSL-код.

### `BSL LS: X errors`

BSL Language Server нашёл ошибки в коде. Это **не блокирует** сборку EPF, но указывает на проблемы.

**Решение**: откройте `.bsl` файл в IDE с BSL LS и исправьте ошибки. Или используйте `--skip-bsl-validation` если уверены в коде.

### `Ошибка: Шаблон не найден`

Шаблоны не найдены в `templates/epf_factory/`.

**Решение**: убедитесь, что вы запускаете из корня репозитория, или что `templates/epf_factory/` существует.

### `v8unpack timeout при сборке`

Сборка заняла более 120 секунд. Возможные причины:
- Очень большой BSL-код
- Проблемы с диском

**Решение**: попробуйте ещё раз, или используйте более короткий BSL-код.

### EPF открывается в 1С с ошибкой

Если 1С ругается на невалидный EPF:
1. Проверьте сигнатуру: первые 4 байта должны быть `ff ff ff 7f`
2. Проверьте round-trip через `--save-sources` и сравните BSL-модули
3. Откройте EPF через `v8unpack -E` и проверьте, что 5 файлов на месте
