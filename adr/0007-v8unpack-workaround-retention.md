# ADR-0007: v8unpack block_size баг — сохранение workaround

**Дата:** 2026-07-04
**Обновлено:** 2026-07-05 (T5.2 — проверка upstream)
**Статус:** Accepted (workaround сохраняется)

## Контекст

v8unpack при сборке .epf пишет `block_size = doc_size` (фактический размер
данных TOC-блока) в заголовке TOC-блока, а 1С ожидает `block_size = 0x200`
(512) для TOC-блока. Из-за этого 1С выдаёт "Ошибка формата потока" при
открытии .epf, собранного v8unpack.

**Уточнение (T5.2, 2026-07-05):** проблема затрагивает ТОЛЬКО TOC-блок.
Data-блоки и description-блоки корректно имеют `block_size = doc_size`
(переменный), как и в реальных EPF от 1С.

Проект содержит workaround: `scripts/patch_epf_blocksize.py` (181 строка),
который находит TOC-блок (сразу после 16-байтного header) и заменяет его
`block_size` на 512. `EpfFactory` (этап 2.2) вызывает этот патчер
автоматически после `v8unpack -B`.

В плане улучшения (задача 3.3) предлагалось 2 варианта:
- A: форкнуть v8unpack, применить патч, publish как v8unpack-forked
- B: дописать нативный парсер .epf на базе cf_extractor.py (333 строки)

## Рассмотренные варианты

### Вариант A: Форкнуть v8unpack

- **Pros:**
  - Решает проблему upstream
  - Другие пользователи v8unpack могут получить фикс
- **Cons:**
  - Требует публикацию на PyPI (отдельная учётка, namespace)
  - Поддержка форка: merge upstream changes, тестирование
  - Solo-dev не может поддерживать 2 пакета
  - v8unpack — сложный пакет (контейнер 1С), тестирование требует реальных .epf
  - Риск regression в форке выше, чем польза

### Вариант B: Нативный парсер .epf

- **Pros:**
  - Полная независимость от v8unpack
  - Eliminate Java-зависимости где возможно
  - Контроль над всем кодом
- **Cons:**
  - Огромная работа: ~2000+ строк (контейнер 1С сложный)
  - Тестирование на 100+ реальных .epf файлах
  - cf_extractor.py (365 строк) — это парсер .cf, а не .epf; нужно расширять
  - Риск regression очень высокий
  - Не вписывается в tempo "один этап за сессию"

### Вариант C: Сохранить workaround (выбран)

- **Pros:**
  - 0 work сейчас
  - Workaround уже работает (1598/1598 тестов)
  - patch_epf_blocksize.py хорошо протестирован (25+ тестов)
  - EpfFactory автоматически применяет патч
  - Документированная проблема (AGENTS.md, ADR-0007, этот документ)
- **Cons:**
  - Зависимость от v8unpack остаётся
  - Workaround может сломаться при обновлении v8unpack (маловероятно — 1.2.6 последняя)
  - "Technical debt" — но documented и isolated

## Решение

**Вариант C — сохранить workaround.**

### Обоснование

1. **Workaround работает:** 1598/1598 тестов проходят, EpfFactory успешно
   создаёт .epf с правильным block_size.

2. **Risk/reward неблагоприятный для A и B:**
   - Форк v8unpack: высокая стоимость поддержки, низкая польза (solo-dev)
   - Нативный парсер: огромная работа, высокий риск regression

3. **v8unpack 1.2.6 — последняя версия:** баг не будет исправлен upstream
   (автор неактивен). Если появится 1.2.7 с фиксом — можно будет убрать
   workaround одной строкой.

4. **Workaround изолирован:** patch_epf_blocksize.py — standalone скрипт,
   не загрязняет основной код. EpfFactory вызывает его через subprocess,
   что позволяет легко отключить.

5. **Документировано:** AGENTS.md содержит правило "всегда применяй патч
   после v8unpack -B". ADR-0007 фиксирует решение.

### Условия для пересмотра

Решение будет пересмотрено при одном из:
- Появится v8unpack 1.2.12+ с фиксом block_size → убрать workaround
- Появится команда для поддержки форка → вариант A
- Покрытие core ≥ 80% и есть время → вариант B (нативный парсер)
- Появятся реальные пользователи с проблемой из-за workaround → пересмотр

## Проверка upstream (T5.2, 2026-07-05)

**Проведена проверка** saby-integration/v8unpack GitHub репозитория:

- **PyPI**: последняя версия — 1.2.6 (на момент проверки 2026-07-05)
- **GitHub main branch**: версия 1.2.11 (5 минорных версий вперёд PyPI)
- **Changelog 1.2.7 → 1.2.11**: НЕТ упоминания фикса block_size
  - 1.2.7: fix расширения с пустой строкой в модуле приложения
  - 1.2.8: fix код возврата при ошибке сборки
  - 1.2.9: new параметр version для расширений и конфигураций
  - 1.2.10: fix панели форм на старых платформах 8.2
  - 1.2.11: new поддержка .erf (ExternalReport), fix OverflowError Windows FILETIME, fix base64 параметров
- **Анализ кода `write_block`**: идентичен в 1.2.6 и 1.2.11 — `min_block_size = max(min_block_size, block_size)` где `block_size = file_size(data) = doc_size` для TOC (когда `total_blocks == 1`)
- **Вывод**: баг block_size для TOC НЕ исправлен в 1.2.11

**Решение T5.2:**
- Обновить зависимость с 1.2.6 до 1.2.11 (через git+https, т.к. 1.2.11 не на PyPI) — полезные фиксы (.erf поддержка, FILETIME fix)
- Workaround `patch_epf_blocksize.py` СОХРАНИТЬ — баг не исправлен
- Native EPF packer (T5.1, M6) остаётся актуальным для устранения зависимости

## Последствия

### Положительные

- ✅ Workaround сохранён, EpfFactory работает
- ✅ 0 regression risk
- ✅ Документированное решение (ADR-0007)
- ✅ Изолированный workaround (standalone скрипт)
- ✅ T5.2: обновление до 1.2.11 даёт поддержку .erf и фиксы (2026-07-05)

### Отрицательные

- ⚠️ Зависимость от v8unpack остаётся (теперь 1.2.11 из GitHub)
- ⚠️ Workaround — technical debt (но documented и isolated)
- ⚠️ При обновлении v8unpack нужно проверить, нужен ли ещё патч
- ⚠️ Зависимость из GitHub (не PyPI) — медленнее установка, зависимость от доступности GitHub

### Future implications

- Задача T5.1 (M6, native EPF packer) остаётся в roadmap — устранит зависимость полностью
- При переходе Beta → Production-Ready можно пересмотреть
- Если community активизируется — можно форкнуть v8unpack вместе
- Если saby-integration опубликует 1.2.11 на PyPI — переключиться обратно на PyPI зависимость
