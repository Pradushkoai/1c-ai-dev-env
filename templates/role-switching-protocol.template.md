# Role-Switching Protocol для задач 1С

> **Обязательный протокол** при решении задач 1С-разработки.
> Полная спецификация: `docs/TASK_PROTOCOL.md`

## 5 ролей

```
🧠 Архитектор     → solve_context, проектирование, план
👨‍💻 Программист   → .bsl код по плану
🎨 Стилист        → check_standards (56 правил) + analyze_bsl (187 диагностик)
📝 Документатор   → docstring + примеры
✅ Проверяющий     → solve_check --level full + verdict
```

## 3 протокола запуска

### Протокол A: Полный (сложные задачи)
**Триггеры:** новая фича, рефакторинг, breaking change, доработка типовой, интеграция

**Роли:** 🧠 Архитектор → 👨‍💻 Программист → 🎨 Стилист → 📝 Документатор → ✅ Проверяющий

### Протокол B: Стандартный (средние задачи)
**Триггеры:** bugfix, небольшая доработка, оптимизация

**Роли:** 👨‍💻 Программист → 🎨 Стилист → ✅ Проверяющий

### Протокол C: Быстрый (тривиальные задачи)
**Триггеры:** опечатка, переименование, версия в Configuration.xml

**Роли:** 👨‍💻 Программист → ✅ Проверяющий (быстрая: только check_standards)

## Алгоритм выбора протокола

```
Получил задачу
    │
    ├─ Задача по проекту 1c-ai-dev-env? → протокол НЕ применяется
    │
    ├─ Задача по 1С (создать объект, .bsl код)?
    │   ├─ Новая фича / рефакторинг / breaking change? → Протокол A
    │   ├─ Bugfix / небольшая доработка? → Протокол B
    │   └─ Опечатка / переименование? → Протокол C
    │
    └─ Вопрос / обсуждение / анализ? → протокол НЕ нужен
```

## Правила работы

1. **Объявляй протокол первым сообщением:**
   ```
   📋 Задача: <формулировка>
   🎯 Протокол: A (Полный) — т.к. <причина>
   Роли: 🧠 → 👨‍💻 → 🎨 → 📝 → ✅
   ```

2. **Объявляй каждую роль перед выполнением:**
   ```
   🧠 [Архитектор] Анализирую...
   → solve_context("запрос", "config")
   ```

3. **Передавай артефакты между ролями:**
   - Архитектор → Программист: план (метаданные, модули, методы, стандарты)
   - Программист → Стилист: .bsl код
   - Стилист → Документатор: стилизованный код
   - Документатор → Проверяющий: документированный код
   - Проверяющий → пользователь: отчёт с verdict

4. **Финальный отчёт Проверяющего:**
   ```
   ✅ [Проверяющий] Финальная проверка
   → solve_check --level full
   → verdict: ready / warnings / errors
   ```

## Критические правила

- ❌ НЕ писать код сразу — сначала Архитектор и `solve_context`
- ❌ НЕ использовать методы платформы по памяти — только из контекста
- ❌ НЕ пропускать роли — даже для быстрой задачи
- ❌ НЕ игнорировать `verdict: errors` — возвращать Программисту
- ❌ НЕ менять логику в роли Стилиста — только стиль
- ✅ Каждая роль объявляется явно
