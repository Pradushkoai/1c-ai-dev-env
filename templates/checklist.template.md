# Чек-лист решения задачи 1С

> **Обязательный чек-лист** при решении любой задачи по 1С-разработке.
> В отличие от «ролей» — это проверяемые условия, а не ритуал.
> Если пункт ❌ — нельзя идти дальше, нужно вернуться.

## Фаза 1: Понимание задачи

- [ ] **ТЗ полное?** Если нет — задать уточняющие вопросы пользователю.
      Не выдумывать реквизиты, методы, формы — только то, что в ТЗ или согласовано.
- [ ] **Конфигурация определена?** `1c-ai config list` — какая конфигурация целевая.
- [ ] **Тип задачи:** новая фича / bugfix / рефакторинг / доработка типовой?

## Фаза 2: Контекст (репозиторий как источник знаний)

- [ ] **Методы платформы:** `solve_context(query, config)` или `1c-ai search "запрос"`
      — получил релевантные методы (не выдумывать по памяти).
- [ ] **API конфигурации:** `get_api_reference(config)` или
      `1c-ai config list` — посмотрел релевантные модули (если применимо).
- [ ] **Стандарты 1С:** понял, какие STD применимы к задаче
      (STD 454 структура модуля, STD 03 запросы, STD 12 клиент-сервер, и т.д.)

## Фаза 3: Реализация

- [ ] **Код написан по ТЗ** — все требования из Фазы 1 учтены.
- [ ] **Методы платформы из контекста** — использованы методы из Фазы 2,
      не выдуманные.
- [ ] **Структура модуля** (если .bsl): 4 области
      (#Область ПрограммныйИнтерфейс / СлужебныйПрограммныйИнтерфейс /
      СлужебныеПроцедурыИФункции / Инициализация).
- [ ] **Запросы** (если есть): через `УстановитьПараметр`, не конкатенация.

## Фаза 4: Проверка (критерии готовности)

- [ ] **`1c-ai standards <file.bsl>`** → 0 errors.
      Если есть errors — **обязательно** исправить, не игнорировать.
      Если error — ложное срабатывание чекера — явно отметить и объяснить.
- [ ] **`1c-ai solve check <file.bsl> --level full`** → verdict ∈ {ready, warnings}.
      Если verdict = errors → **вернуться к Фазе 3** и исправить.
- [ ] **BSL LS** (`1c-ai bsl analyze`) → 0 errors (warnings допустимы).

## Фаза 5: Документация

- [ ] **Docstring** для всех экспортных процедур/функций:
      Назначение, Параметры, Возвращаемое значение, Пример.
- [ ] **Примеры использования** — если задача создаёт новый программный
      интерфейс (функции/процедуры Экспорт).
- [ ] **Описание изменений** для пользователя — что сделано, как использовать.

## Финальная проверка

- [ ] **Все пункты выше ✓** (или явно обоснованные исключения).
- [ ] **Результат передан пользователю** с краткой сводкой.

---

## Anti-patterns (что НЕ делать)

1. **Не начинать с `solve_context` без понимания ТЗ.** Сначала уточни задачу.
2. **Не выдумывать реквизиты/методы/формы.** Только то, что в ТЗ или в контексте.
3. **Не игнорировать `verdict: errors`.** Возвращаться к реализации.
4. **Не имитировать фазы.** Если пункт ❌ — остановись и исправь.
5. **Не генерировать «демо-код».** Код должен решать реальную задачу.

---

## Пример использования

```
Пользователь: "Создай справочник ТоварыСемена в УТ11"

Я:
  Фаза 1:
    □ ТЗ полное? ❌ — не сказано, какие реквизиты, иерархия, формы
    → СПРОСИТЬ: "Какие реквизиты? Иерархия? Формы нужны?"
    (НЕ генерировать код, пока не ответит)

  Фаза 2: (после получения ТЗ)
    □ solve_context("создать справочник", "ut11") — получил СоздатьЭлемент, НайтиПоКоду
    □ get_api_reference("ut11") — посмотрел CommonModules

  Фаза 3:
    □ Код по ТЗ (с реквизитами от пользователя)
    □ Методы из контекста
    □ Структура модуля

  Фаза 4:
    □ 1c-ai standards module.bsl → 0 errors ✓
    □ 1c-ai solve check --level full → verdict=warnings ✓

  Фаза 5:
    □ Docstring для экспортных
    □ Примеры
    □ Сводка пользователю
```
